# AI 判定卡片设计 v1.1

> 客服工单判定的核心交互工具
> 设计目标：让客服 30 秒内做出决定，70% 案件 AI 自动建议、客服一键确认

---

## 0. 设计背景

### 当前问题
- 客服处理一单工单平均 5–10 分钟
- 50% 工作量集中在"买家付款 / 卖家不发货"
- 客服需要在多个 Tab 间切换：付款截图、聊天记录、用户历史、银行后台
- 决策依赖个人经验，质量参差

### 目标状态
- 客服 30 秒内完成简单案件
- AI 给出"建议判定 + 置信度 + 理由链"
- 简单案件（高置信度）→ 客服一键确认
- 复杂案件 → AI 把所有证据预先聚合，客服仅做判断

## 2. 数据输入

### 2.1 用户提交的证据

| 类型 | 来源 | 处理方式 |
|---|---|---|
| 付款截图 | 买家上传 | OCR + 图像哈希查重 |
| 聊天记录 | App 内 IM | 文本提取 + 关键词识别 |
| 银行短信 | 买家上传（可选）| OCR + 短信指纹库比对 |
| 视频证据 | 买家上传（少见）| 抽帧 + 多模态分析 |

### 2.2 系统侧自动拉取

| 类型 | 来源 | 用途 |
|---|---|---|
| 订单创建/付款/放币时间戳 | 交易系统 | 时效判断 |
| 卖家预绑定收款账户 | 用户中心 | 收款方匹配 |
| 双方信用画像 | 信用体系（见 [信用文档]）| 信任度评估 |
| 双方近 90 天交易历史 | 交易系统 | 行为模式 |
| Sybil 关联组 | 反作弊系统 | 串谋检测 |
| 工单类型/聊天关键词 | NLP 模型 | 案件分类 |

### 2.3 第三方对接（可选）

| 类型 | 来源 | 触发条件 |
|---|---|---|
| 银行流水 API | 网联 / 银联 / 第三方 | 卖家授权后调用 |
| 链上转账查询 | 区块链浏览器 | USDT 链上确认 |
| 公安联网征信 | 实名查询接口 | 重大案件升级时 |

## 3. AI 推理流程

```
工单创建
    ↓
数据聚合层（并发）
    ├─ OCR 服务（付款截图 / 短信）
    ├─ 用户画像查询（信用体系）
    ├─ 交易号查重（防截图复用）
    ├─ 银行流水比对（如已对接）
    └─ Sybil 检测（关联账号）
    ↓
信号汇聚
    ↓
LLM 推理（GPT-5 / Claude 4.6）
    输入：所有结构化证据 + 用户画像 + 业务规则
    输出：建议判定 + 置信度 + 理由链
    ↓
置信度路由
    ├─ ≥ 90% → AI 直接生成判定卡片，客服一键确认
    ├─ 70–90% → AI 生成卡片，客服必须查看证据后确认
    └─ < 70% → 转人工复核，AI 仅做证据聚合
```

## 4. 信用体系如何 Feed In

这是 AI 判定卡片和信用体系最关键的联动。

### 4.1 双方信用快照（每次工单触发时锁定）

```json
{
  "buyer": {
    "user_id": "user_8821",
    "level": "L3",
    "credit_score": 612,
    "register_days": 312,
    "stats_90d": {
      "success_count": 142,
      "fail_count": 1,
      "ticket_count": 0,
      "on_time_pay_rate": 0.98
    },
    "sybil_group_id": null
  },
  "seller": {
    "user_id": "user_5503",
    "level": "L4",
    "credit_score": 712,
    "register_days": 521,
    "stats_90d": {
      "success_count": 480,
      "fail_count": 3,
      "ticket_count": 2,
      "on_time_release_rate": 0.89
    },
    "sybil_group_id": null,
    "permanent_ban_strikes": 0
  }
}
```

### 4.2 信用差对判定的影响

| 双方信用差 | AI 倾向 | 阈值调整 |
|---|---|---|
| 买家 L4+ vs 卖家 L1 | 偏向买家 | 置信度 ≥ 70% 即可建议 |
| 买家 L1 vs 卖家 L4+ | 偏向卖家 | 置信度需 ≥ 90% 才建议 |
| 双方等级相近 | 中立 | 凭证据 |

→ **不是"高等级一定赢"，而是"高等级用户的证据可信度更高"**。

### 4.3 历史工单率作为权重

```
seller.ticket_rate_90d = 工单数 / 总订单数

if ticket_rate > 5%：
    AI 在所有判定中减少对该卖家的可信度权重 -20%
if ticket_rate > 10%：
    AI 自动建议人工介入，不再给出建议
```

### 4.4 Sybil 关联

```
if buyer.sybil_group == seller.sybil_group：
    AI 标记 "可疑串谋"
    建议：暂缓判定 + 风控复审
```

## 5. 置信度算法

### 5.1 多信号加权

```
confidence = w1 * ocr_match_score
           + w2 * bank_flow_match_score
           + w3 * counterparty_credit_score
           + w4 * counterparty_behavior_score
           - w5 * red_flag_penalty

权重示例：
w1 = 0.30  OCR 与系统数据匹配度
w2 = 0.30  银行流水匹配度（如已对接）
w3 = 0.15  双方信用差
w4 = 0.15  双方行为画像
w5 = 0.10  风险信号（Sybil、关联账号、异常等）
```

### 5.2 红旗信号（直接降低置信度）

| 信号 | 降低置信度 |
|---|---|
| 付款截图交易号已被复用 | -50%（直接转人工）|
| 收款方与卖家预绑定账户不符 | -40% |
| 双方在 Sybil 关联组 | -30% |
| 卖家 30 天内有 ≥ 3 次工单退币 | -20% |
| OCR 置信度 < 80% | -15% |

## 6. 自动判定规则

### 6.1 自动通过（无需客服）

```
当满足全部条件时：
- AI 置信度 ≥ 95%
- 双方信用都 ≥ L3
- 无任何红旗信号
- 案件类型属于"标准型"（非异常）
- 涉及金额 < 50,000 元

→ 系统直接执行 AI 判定，仅事后审计
```

> 上线初期建议关闭此规则，T+90 后开启。

### 6.2 客服一键确认

```
当满足条件时：
- AI 置信度 ≥ 90%
- 客服查看卡片后无异议

→ 一键确认，AI 自动生成判定理由
```

### 6.3 人工深度介入

```
当任一条件触发：
- AI 置信度 < 70%
- 红旗信号触发
- 涉及金额 ≥ 100,000 元
- 涉及 Sybil 关联
- 用户在 30 天内有 3+ 次工单

→ 进入人工深度审理流程
```

## 7. 客服反馈闭环

```
客服每次操作（确认 / 修改 / 驳回 AI 建议）
    ↓
反馈数据入库
    ↓
每日聚合分析：AI 建议准确率
    ↓
低准确率案件类型 → 标记为"需要 fine-tune"
    ↓
每月模型迭代 / Prompt 优化
```

### 关键指标

```
AI 建议采纳率（客服一键确认占比）
AI 建议准确率（事后无申诉的占比）
客服平均处理时长
工单平均生命周期
```

## 8. 防滥用 / 公平性

### 8.1 不让 AI 永远站强者一边

- 高等级用户也会犯错。AI 必须基于"证据"而非"身份"
- 信用画像只是"权重"，不是"决定因素"
- 客服 override AI 建议时，记录 override 理由 → 反向训练

### 8.2 透明度

- 每个判定都有完整理由链（推理 1→2→3→...）
- 用户申诉时可查看 AI 推理过程
- 法务/合规可追溯每条决策

### 8.3 不可解释的不输出

- 如果 LLM 给不出清晰的理由链 → 直接转人工
- 不允许"我觉得应该 X 但说不清原因"

## 10. 上线节奏（与信用体系对齐）

| 阶段 | AI 卡片状态 |
|---|---|
| T+0 | 卡片上线，仅做"证据聚合"，不给建议 |
| T+15 | 开放 AI 建议，但客服 100% 必须确认 |
| T+30 | 高置信度（≥ 90%）案件支持"一键确认" |
| T+60 | 启用客服反馈闭环，开始模型 fine-tune |
| T+90 | 探索"自动判定"（小金额 + 高置信度）|
| T+180 | 全面评估，决定是否扩大自动判定范围 |
