# 用户登录功能 - 测试用例

## 测试概要
- **测试范围**：
  - **P0 核心功能**：手机号 OTP 登录、JWT 登录态管理、基础安全防护（频控、错误次数限制）、登录日志记录。
  - **P1 扩展功能**：邮箱 OTP 登录、账号密码登录、设备记忆、退出所有设备。
  - 覆盖前端 UI 交互、后端 API 接口逻辑、数据存储正确性。
- **前置条件**：
  - 测试环境已部署最新的前端和后端代码。
  - 数据库中已准备好不同状态的测试账号（如：正常、已锁定、未注册手机号、未设置密码等）。
  - OTP (短信/邮件) 服务在测试环境下可用，并可查询发送记录。
  - 测试人员拥有访问后端日志和数据库的权限，以便验证结果。
- **测试环境要求**：
  - 后端服务：Test 环境。
  - 前端应用：部署在 Test 环境。
  - 浏览器：Chrome, Safari, Firefox 最新版。
  - 数据库：Test 环境数据库。
  - 工具：API 测试工具 (如 Postman), 浏览器开发者工具。

## 安全测试用例

| 编号 | 安全场景 | 测试方法 | 预期结果 | 优先级 |
|------|----------|----------|----------|--------|
| ST-001 | **XSS 注入** | 在手机号、邮箱、用户名等所有输入框中输入 `<script>alert('xss')</script>`。 | 输入被正确转义，脚本不会被执行，页面上仅显示为纯文本。 | P0 |
| ST-002 | **SQL 注入** | 在用户名和密码字段输入 `' OR '1'='1' --` 等 SQL 注入 payload。 | 登录失败，提示“账号或密码错误”。后端使用参数化查询，无 SQL 注入漏洞。 | P0 |
| ST-003 | **越权访问** | 1. 用户 A 登录。<br>2. 尝试调用获取其他用户信息的接口（如 `/api/user/{user_B_id}/...`）。 | 服务器返回 403 Forbidden 或 404 Not Found，用户 A 无法获取用户 B 的数据。 | P0 |
| ST-004 | **暴力破解防护** | 使用脚本对单个账户进行高频次的密码猜测。 | 在 5 次尝试后，该账户被锁定，后续请求在 10 分钟内均失败。IP 也可能被临时拉黑。 | P0 |
| ST-005 | **凭证存储安全** | 1. 登录成功后，打开浏览器开发者工具。<br>2. 查看 Application -> Cookies。 | JWT (Access Token) 应存储在 `HttpOnly` 和 `Secure` 标记的 Cookie 中，防止被 JS 读取。 | P0 |
| ST-006 | **敏感信息泄露** | 检查所有 API 的响应体，包括错误响应。 | 响应中不应包含如数据库密码、堆栈跟踪、服务器版本等敏感信息。错误提示应是通用的。 | P0 |
| ST-007 | **CSRF 防护** | 尝试构造一个恶意页面，其中包含一个指向“退出登录”或“退出所有设备”接口的表单或请求。 | 由于缺少 CSRF Token 或未使用 SameSite Cookie 策略，请求失败，用户不会被恶意退出。 | P1 |
| ST-008 | **重放攻击 (OTP)** | 1. 拦截一个使用过的、有效的 OTP 登录请求。<br>2. 在该 OTP 过期或被使用后，重新发送此请求。 | 服务器应拒绝该请求，因为 OTP 是一次性的，已被消费或已过期。 | P0 |
| ST-009 | **密码存储安全** | （需后端配合或数据库权限）检查 `users` 表中的 `password_hash` 字段。 | 密码必须经过加盐哈希（如 bcrypt, scrypt, argon2）存储，绝不能是明文或简单的 MD5。 | P0 |

## 性能测试建议

| 指标 | 基准要求 | 测试方法 |
|------|----------|----------|
| OTP/密码登录接口 | 95% 响应时间 < 200ms | 使用 JMeter 或 k6，模拟用户登录流程，逐步增加并发用户数，监控接口响应时间。 |
| 并发登录 | 支持 100 TPS (Transactions Per Second) | 使用 JMeter 或 k6，设置 100 并发线程，持续发起登录请求，观察系统成功率和响应时间。 |
| OTP 发送接口 | 95% 响应时间 < 500ms | 压测 `/api/auth/otp/send` 接口（注意遵守频控规则或在测试时临时放开），监控接口性能。 |
| 页面加载性能 | 首次可交互时间 (TTI) < 3s | 使用 Lighthouse 或 WebPageTest 对登录页面进行分析，优化前端资源加载。 |

## 回归测试清单
本次“用户登录”功能是系统的核心基础模块，上线或变更将影响所有依赖用户认证的功能。
- **必须回归的范围**：
  1. **所有受保护的页面**：确保未登录用户无法访问，登录后可以正常访问。
  2. **用户个人资料/账户设置页面**：确保能正确加载当前登录用户的信息。
  3. **下单、交易等核心业务流程**：确保用户身份在整个流程中被正确识别和传递。
  4. **权限管理系统**：如果系统有角色和权限，需验证不同角色的用户登录后，其权限是否正确。
