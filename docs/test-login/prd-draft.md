# 用户登录功能 - PRD 草案（PM-Alpha）

## 需求背景
当前系统缺少/需要验证「用户登录」能力，用于：
- 识别用户身份，建立会话状态（Session/Token）
- 支撑后续受保护操作（查看资产、下单、提币、风控校验等）
- 为安全能力打底（设备识别、异常登录、二次验证）

本需求为“测试需求”，但我们仍要把登录做成可扩展的地基：不仅能登录，还能为未来风控、增长、合规预留接口。我们能不能比竞品做得完全不一样？可以——把登录从“填表”升级成“无感且更安全”。

## 目标用户
- 新用户：首次访问，需要注册/登录后才能使用核心功能
- 老用户：高频访问，需要快速登录、少打扰
- 风控敏感用户：异地/新设备/高风险行为触发更强校验
- 测试/运营人员：需要可控的测试账号体系与可观测性

## 方案对比

### 方案 A：账号密码 + 短信/邮箱验证码（行业标准做法）
- 核心功能点
  - 登录方式：手机号/邮箱 + 密码
  - 可选二次验证：短信验证码或邮箱验证码（开关可配）
  - 基础安全：失败次数限制、验证码冷却、简单黑名单
  - 会话管理：登录态（JWT 或 Session），退出登录，token 刷新
  - 找回密码：短信/邮箱重置
  - 管理后台支持：封禁账号、重置密码、查看登录日志（基础）

- 用户故事
  - 作为用户，我输入手机号和密码即可登录；如果忘了密码可以通过验证码找回。
  - 作为系统，我在多次失败后限制登录，防止撞库。

- 创新亮点
  - 作为基线方案，优势在于：实现快、用户认知成熟、测试覆盖容易。

- 潜在风险
  - 密码带来高维护成本：撞库、弱密码、客服量
  - 短信成本与可用性（延迟/拦截/海外不可达）
  - 用户体验偏“表单化”，登录摩擦大

---

### 方案 B：无密码登录（OTP/魔法链接）+ 设备记忆 + 风险分级（创新优化方案）
- 核心功能点
  - 登录方式优先：短信/邮箱 OTP（一次性验证码）或邮箱魔法链接（Magic Link）
  - 设备记忆：可信设备 30 天免二次验证（可配置）
  - 风险分级策略（Risk-based Authentication）
    - 低风险：直接 OTP 登录
    - 中风险（新设备/异地/IP异常）：追加图形验证码或二次 OTP
    - 高风险：强制二次验证 + 冷却期 + 通知提醒
  - 登录提醒：新设备登录邮件/短信通知（含一键冻结入口）
  - 可观测性：埋点与审计字段（登录耗时、OTP到达率、失败原因分布）

- 用户故事
  - 作为用户，我不想记密码；输入手机号，收验证码，30 秒内登录。
  - 作为老用户，我常用同一台设备，之后可以“更快、更少打扰”地进入系统。
  - 作为风控系统，我检测到新设备登录，会提高校验强度并通知用户。

- 创新亮点
  - 把“安全”做成“分级体验”：大多数时候丝滑，少数高风险时才加严
  - 显著降低密码相关风险与客服成本
  - 更利于增长漏斗：减少“忘记密码/密码错误”流失

- 潜在风险
  - OTP 依赖短信/邮箱通道稳定性（需要供应商与重试策略）
  - 风险策略需要持续调参，否则会误伤（过严影响转化，过松影响安全）
  - 需要更完善的日志/监控，否则难定位“收不到验证码”等问题

---

### 方案 C：钱包签名一键登录（Web3 Sign-in）+ “游客态→秒升级”合并账户（颠覆性方案）
> 我们能不能比竞品做得完全不一样？可以：把登录从“验证你是谁”改成“证明你控制某个密钥”。对 USDT 相关产品尤其契合。

- 核心功能点
  - 登录方式：连接钱包（如 MetaMask/WalletConnect）→ 请求签名 → 验证签名 → 登录成功
  - 支持游客态：未登录也可浏览核心页面/模拟操作；需要关键操作时再弹出签名登录（渐进式登录）
  - 账户体系：以钱包地址为主键；支持绑定手机号/邮箱作为找回/通知通道（非必须）
  - 安全增强：
    - 签名挑战（nonce）防重放
    - 设备指纹 + 风险提示（可选）
  - 迁移合并：若用户先用手机号登录，后接入钱包，可合并资产/权限归属（需规则）

- 用户故事
  - 作为用户，我不想填任何信息；点“连接钱包”，签名一次就进入系统。
  - 作为新用户，我先以游客浏览，确定要使用时再一键升级为登录态，不被注册表单打断。
  - 作为系统，我通过签名证明用户对地址的控制权，从而建立身份与权限。

- 创新亮点
  - 登录体验极简：无密码、无验证码（在钱包可用的前提下）
  - 与 USDT/交易语境天然一致：用“密钥所有权”替代“账号所有权”
  - 渐进式登录显著提升转化：先体验价值，再做身份确认

- 潜在风险
  - 受钱包生态影响：用户不一定有钱包/不熟悉签名
  - 合规/风控要求可能仍需 KYC/手机号等补充信息（需预留流程）
  - 签名交互若设计不好会造成“钓鱼感”（必须清晰文案与安全提示）
  - 移动端兼容与钱包唤起链路复杂，需要专项测试

## 推荐方案及理由
**推荐方案：B（无密码登录 + 设备记忆 + 风险分级）**，原因：
1. **落地速度与可靠性最好**：相较方案 C 生态依赖更少；比方案 A 更现代、更少账号安全负担。
2. **体验与安全不对立**：用风险分级把“更安全”集中在少数场景，把“更顺滑”给大多数用户。
3. **可平滑演进到 C**：未来可在 B 的登录入口旁增加“钱包一键登录”，作为增量渠道，而不推翻体系。

> 激进但务实：先用 B 把“登录地基 + 风控接口”铺好，再把 C 当作差异化武器逐步上线。

## 用户故事（推荐方案详细版）
1. 新用户首次登录（低风险）
   - 用户输入手机号/邮箱 → 点击获取验证码
   - 系统发送 OTP（含重试与冷却）
   - 用户输入 OTP → 登录成功 → 进入首页
2. 老用户常用设备快速登录
   - 用户输入手机号/邮箱 → 获取 OTP → 登录
   - 系统提示“记住此设备 30 天”默认开启
   - 下次登录若设备可信且风险低，可减少额外校验步骤（仍需 OTP；或可配置为更短流程）
3. 新设备/异地登录（中高风险）
   - 用户输入手机号/邮箱 → 获取 OTP
   - 系统判断为新设备/异常 IP → 追加校验（例如图形验证码/二次 OTP/延迟冷却）
   - 登录成功后发送“新设备登录提醒”，含“一键冻结/退出所有设备”
4. 安全自助
   - 用户在“安全设置”查看登录设备列表、最近登录记录
   - 支持“退出其他设备/全部退出”
5. 运营/测试故事（可观测）
   - 测试人员可在日志中定位：OTP 发送成功率、到达耗时、失败原因（过期/错误/频控/风控拦截）

## 初步验收标准
1. 基础登录
   - 支持手机号或邮箱发起登录
   - OTP 验证成功后创建登录态（JWT/Session），可访问受保护接口
   - OTP 具备：有效期（如 5 分钟）、错误次数限制（如 5 次）、发送频控（如 60 秒冷却）
2. 设备记忆
   - 同一设备在配置期限内标记为可信（可开关、可配置天数）
   - 可在安全设置中查看并移除可信设备
3. 风险分级（最小可用版本）
   - 至少识别：新设备（无设备标识）、异地/异常 IP（规则可简化）
   - 触发风险时有明确提示与额外校验手段（先用图形验证码或更严格频控也可）
4. 通知与审计
   - 新设备登录成功后发送通知（邮箱/短信二选一即可）
   - 记录登录日志：时间、IP、设备信息、结果、失败原因
5. 退出与会话
   - 支持退出登录
   - 支持“退出所有设备”（服务端使 token 失效或变更会话版本号）
6. 可测试性
   - 提供测试环境开关/白名单策略（例如固定 OTP 或可控模拟通道）以便自动化测试
