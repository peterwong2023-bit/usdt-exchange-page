# 用户登录功能 - PRD（定稿）

## 需求背景
当前系统需要验证「用户登录」能力，用于：
- 识别用户身份，建立会话状态
- 支撑后续受保护操作（查看资产、下单等）
- 为基础安全能力打底

**务实原则：** 本期目标是快速上线可用的登录功能，优先保证稳定性和可测试性。

## 目标用户画像
- **主要用户：** 测试人员和内部用户，需要快速、稳定的登录体验
- **次要用户：** 未来的真实用户，为后续优化预留空间

## 功能清单

### P0 - 必须有
| 功能 | 描述 | 验收标准 |
|------|------|----------|
| 手机号OTP登录 | 输入手机号→获取验证码→输入验证码→登录成功 | OTP 5分钟有效期，支持重发（60s冷却） |
| 登录态管理 | JWT Token 管理，支持登录/退出 | Token 有效期7天，支持刷新机制 |
| 基础安全防护 | 验证码错误次数限制、发送频控 | 5次错误锁定10分钟，发送间隔60秒 |
| 登录日志 | 记录登录时间、IP、设备信息、结果 | 支持后台查询和导出 |

### P1 - 应该有  
| 功能 | 描述 | 验收标准 |
|------|------|----------|
| 邮箱OTP登录 | 邮箱作为备用登录方式 | 与手机号逻辑一致 |
| 账号密码登录 | 传统登录方式作为backup | 密码8位以上，支持找回密码 |
| 设备记忆 | 可信设备30天内减少验证步骤 | 可在设置中管理设备列表 |
| 退出所有设备 | 批量使Token失效 | 变更会话版本号，所有设备需重新登录 |

### P2 - 锦上添花
| 功能 | 描述 | 验收标准 |
|------|------|----------|
| 登录通知 | 新设备登录邮件/短信提醒 | 包含设备信息和异常举报入口 |
| 简单风控 | IP异常检测 | 异地登录增加图形验证码 |
| 社交登录 | Google/Apple第三方登录 | OAuth2.0标准实现 |

## 用户故事（按角色分组）

### 作为测试人员
- 我需要快速登录测试环境，验证其他功能
- 我需要能够重置登录状态，测试各种场景
- 我需要查看详细的登录日志，排查问题

### 作为产品用户
- 我输入手机号获取验证码即可登录，过程在1分钟内完成
- 我在常用设备上下次登录更便捷
- 我能看到账号的登录记录，确保安全

### 作为运营人员
- 我能够查看用户登录数据，分析登录成功率
- 我能够处理用户登录问题的客服请求
- 我能够管理异常账号（封禁/解封）

## MVP 范围

### v1.0 只做什么
- 手机号OTP登录（核心流程）
- JWT Token会话管理
- 基础安全防护（频控、次数限制）
- 简单的后台管理（查看日志、重置密码）

### v1.0 不做什么  
- 复杂的风险控制策略
- 钱包签名登录
- Magic Link邮件登录
- 高级的设备指纹识别
- 多地域短信服务商容灾

## 验收标准

### 功能验收
1. **登录流程：** 正常情况下30秒内完成登录
2. **安全验收：** 通过基础的渗透测试（暴力破解、重放攻击）
3. **性能验收：** 支持100并发用户同时登录
4. **稳定性验收：** OTP发送成功率>95%

### 技术验收
1. **API响应：** 登录接口平均响应时间<200ms
2. **数据库：** 登录日志完整记录，支持按时间/用户查询
3. **监控：** 关键指标可观测（登录成功率、OTP到达率、错误分布）

## PM 辩论记录

| 议题 | PM-Alpha 观点 | PM-Beta 观点 | 最终决策 | 理由 |
|------|---------------|---------------|----------|------|
| 主要登录方式 | 推荐无密码OTP + 风险分级 | 同意OTP，但简化风险策略 | OTP为主，简化风控 | 测试需求优先稳定性 |
| 钱包登录 | 作为颠覆性创新方案 | 砍掉，过于超前 | 砍掉 | 技术依赖重，测试阶段不必要 |
| 设备记忆 | P0核心功能 | 降级为P1 | P1实现 | 不是登录最小可用功能 |
| Magic Link | 作为OTP补充方案 | 砍掉，增加复杂度 | 砍掉 | 邮件服务依赖，测试环境下收益小 |
| 风险分级策略 | 完整的RBA系统 | 简化为基础IP检测 | 简化版本 | 避免误伤，运营成本可控 |
| 渐进式登录 | 游客态→秒升级创新体验 | 砍掉，增加状态复杂度 | 砍掉 | 测试需求不需要转化优化 |
