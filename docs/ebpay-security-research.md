# EBpay 钱包市场调研 — 安全体系分析与防假客服方案

> 调研日期: 2026-03-09
> 调研范围: EBpay 钱包 App 安全体系、用户层安全技术、防假客服产品方案

## 一、EBpay 产品概述

EBpay 是一款 **加密货币支付钱包 + 商户 USDT 收款网关** 双重定位的产品：

- **C 端**：通过自有 EB 代币（1:1 锚定法币）实现法币与数字货币双向兑换
- **B 端**：点对点 (P2P) USDT 收款支付网关，非托管模式
- 支持币种：USDT、USDC、BTC、ETH、TRX、BNB 等
- 法币入金：银行卡、微信、支付宝、数字人民币、银联

## 二、EBpay 安全体系全景

### 2.1 安全架构分层

```
用户层
├── KYC 实名认证（身份证正反面 + 证件签发国家选择）
├── 生物识别（指纹 / Face ID，调用 OS 原生 Secure Enclave / TEE）
├── Google Authenticator 2FA（TOTP，RFC 6238，32位 Base32 密钥，30秒周期）
├── SMS 短信验证码 + 邮箱验证码
├── 登录密码 + 独立交易密码（双密码分离）
└── 银行卡实名绑定（与 KYC 身份一致性校验）

传输层
├── SSL 双向认证
└── 全链路动态加密

应用层
├── 行为风控引擎（实时异常数据监控）
├── USDT 地址风控（5 级评级：无风险/低/中/高/重大，链上溯源）
├── 防撞单系统（自动生成新地址 + 订单金额浮动）
└── P2P 撮合（普通匹配 / 极速匹配）

存储层
├── AES-256 数据加密
└── 区块链分布式存储（宣称）

运维层
├── 每日安全扫描
├── 白帽渗透测试
├── 第三方安全审计
└── 云原生 + 灰度发布 + 回滚机制
```

### 2.2 用户层安全深度分析

#### Google Authenticator (TOTP 2FA)

| 维度 | EBpay 实现 | 行业最佳实践 |
|------|-----------|-------------|
| 算法 | 标准 RFC 6238 TOTP | 相同 |
| 密钥长度 | 32 位 Base32 (160bit) | 通常 160-256bit，达标 |
| 绑定流程 | 三重验证（手机/邮箱 + 谷歌码 + 交易密码） | 优秀 |
| 备份恢复码 | 未明确提供 | Binance/OKX 提供一次性恢复码 |
| 硬件安全密钥 | 不支持 | Coinbase 支持 FIDO2/WebAuthn |
| 反钓鱼码 | 未提供 | Binance 提供邮件反钓鱼码 |

#### 生物识别

- 调用 iOS `LocalAuthentication` / Android `BiometricPrompt` 原生 API
- 密钥存储在 Secure Enclave (iOS) / TEE+StrongBox (Android)
- 生物特征转化为加密数学模板，不存储原始数据（OS 层保证）
- **未明确**：生物识别是仅用于 App 解锁，还是每笔交易签名

#### KYC 实名认证

| 维度 | EBpay 现状 | 行业标准 |
|------|-----------|---------|
| 证件上传 | 身份证照片 | OCR 自动识别 + 证件真伪校验 |
| 活体检测 | **未明确** | 眨眼/点头/张嘴等活体检测 |
| 人脸比对 | **未明确** | 证件照与实时人脸 1:1 比对 |
| AML 筛查 | 未提及 | 制裁名单/PEP 数据库筛查 |

#### 密码体系

- 登录密码与交易密码分离（金融行业标准做法）
- 密码强度策略、存储哈希算法（Argon2/bcrypt）、暴力破解防护细节不明

### 2.3 EBpay 安全缺失项

| 安全机制 | 状态 | 影响 |
|---------|------|------|
| 活体检测 KYC | 缺失 | 身份伪造风险 |
| 反钓鱼码 | 缺失 | 多域名运营加剧钓鱼风险 |
| 硬件安全密钥 (FIDO2) | 缺失 | 高价值账户无最强认证手段 |
| 设备管理面板 | 未提及 | 用户无法查看/管理已登录设备 |
| 异地登录告警 | 未提及 | 账户被盗时无即时感知 |
| 防截屏/防录屏 | 未提及 | 敏感信息可能泄露 |
| 越狱/Root 检测 | 未提及 | 不安全环境下运行风险 |
| 剪贴板安全 | 未提及 | 地址可能被剪贴板劫持篡改 |
| 提币地址白名单 | 未知 | 无法限制只向信任地址转账 |

### 2.4 风险评估矩阵

| 攻击向量 | 风险 | 原因 |
|---------|------|------|
| 密码暴力破解 | 🟡 中 | 有 2FA 但密码策略细节不明 |
| SIM Swap 攻击 | 🔴 高 | SMS 验证码可被 SIM 卡劫持 |
| 钓鱼攻击 | 🔴 高 | 无反钓鱼码，多域名增加混淆 |
| 生物特征伪造 | 🟢 低 | 依赖 OS 层 Secure Enclave |
| KYC 身份伪造 | 🟡 中 | 无活体检测证据 |
| 设备丢失/被盗 | 🟡 中 | 有生物识别但 2FA 恢复机制不明 |
| 中间人攻击 | 🟢 低 | SSL 双向认证 |
| 假客服盗号 | 🔴 高 | 无官方验证机制，多域名运营 |

## 四、防假客服盗号方案（产品设计）

### 4.1 攻击链分析

```
假客服典型攻击流程:

1. 潜入  → 诈骗者混入官方 Telegram / 微信 / QQ 群
2. 伪装  → 头像、昵称、简介完全模仿官方客服
3. 触发  → 等用户在群里提问或报错
4. 私聊  → 主动私信用户："我是官方客服，帮你处理"
5. 诱导  → 发送假链接 / 要求提供账号密码 / 要求转账"验证"
6. 得手  → 获取密码、验证码，或直接骗走资金

用户上当原因:
  ✗ 无法验证对方身份
  ✗ 问题紧急时判断力下降
  ✗ "客服"外观太像真的
```

### 4.2 方案一：App 内唯一客服通道（P0 优先级）

**核心原则：官方永远不会在第三方平台主动私聊用户**

```
App 首页 / 个人中心
│
└── [💬 在线客服] ← 唯一官方客服入口
    │
    ├── 安全提示:
    │   "EBpay 客服只在此页面沟通。
    │    任何在 Telegram/微信/QQ 主动联系你的都是骗子"
    │
    └── 会话验证码:
        每次客服会话生成临时验证码（如 A7X92K）
        客服需先报出此验证码证明身份
        所有对话记录留存可追溯
```

**效果**：从根本上消除第三方假客服问题。

### 4.3 方案二：敏感操作冷静期 + 勾选声明（P0 优先级）

```
触发条件:
  - 首次向新地址转账
  - 大额交易（超过用户平均交易额 3 倍）
  - 短时间内多次修改安全设置

交互设计:
┌──────────────────────────────────────┐
│ ⚠️ 安全确认                          │
│                                      │
│ 您正在向新地址转账 5000 USDT          │
│                                      │
│ 请确认以下信息:                       │
│ □ 这不是客服要求我做的操作             │
│ □ 没有人在远程指导我操作               │
│ □ 我确认这是我本人的自主操作           │
│                                      │
│ ⏱️ 请仔细阅读 (15秒后可确认)          │
│                                      │
│        [取消交易]                     │
└──────────────────────────────────────┘

设计要点:
  - 强制用户逐条勾选声明，而非一键确认
  - 加入 15-30 秒强制等待时间，打断冲动决策
  - 老地址 / 白名单地址可跳过
```

### 4.4 方案三：屏幕共享/远程控制检测（P1 优先级）

```
检测维度                          触发动作
─────────────────────────────────────────────────────────
屏幕共享/远程控制软件运行中      → 弹窗警告 + 阻止敏感操作
(AnyDesk/TeamViewer/向日葵等)     "检测到您正在使用屏幕共享，
                                   为安全起见已暂停交易功能"

通话中执行敏感操作                → 弹窗警告
                                   "检测到您正在通话中操作，
                                    请警惕电话诈骗"

剪贴板地址与输入地址不一致        → 地址篡改提醒
                                   "您粘贴的地址与剪贴板内容不同，
                                    可能存在恶意软件"

短时间内多次修改安全设置          → 临时锁定 + 人工审核
```

**这是最高效的拦截手段** — 大量假客服要求用户开启屏幕共享，检测到即锁死资金操作。

### 4.5 方案四：官方人员验证系统（P2 优先级）

```
场景: 用户在群聊遇到声称是官方的人

App 内功能: "验证官方身份"
  1. 让对方提供"员工验证码"
  2. 打开 App → 安全中心 → 验证官方身份
  3. 输入验证码

结果:
  ✅ 已验证: 客服 张XX (工号 CS-0042)
     头像: [真实头像]
     有效期至: 2026-03-09 15:30

  ❌ 验证失败: 该验证码无效
     ⚠️ 对方可能是假冒客服，请立即停止沟通！

技术实现:
  - 客服端生成 TOTP 动态验证码（类似 Google Authenticator）
  - 用户在 App 内查询验证，结果展示客服真实信息
  - 验证码 5 分钟有效，防止截获重放
```

### 4.6 方案五：社群官方 Bot 验证（P2 优先级）

```
Telegram 群内部署官方 Bot:

用户: @EBpay_bot 验证 @某某客服

Bot 回复:
  ❌ @某某客服 不是 EBpay 官方人员
  ⚠️ 请勿向其提供任何个人信息！

  ✅ EBpay 官方客服列表:
     @EBpay_Support_01 (已认证)
     @EBpay_Support_02 (已认证)

  🔒 官方客服永远不会主动私信你

配合群设置:
  - 禁止非管理员私聊群成员
  - 新用户入群时 Bot 自动发防诈骗提醒
```

### 4.7 方案六：用户教育融入产品（P1 优先级）

| 触点 | 做法 |
|------|------|
| App 首次启动 | 强制观看 30 秒防骗视频 |
| 每次打开客服页面 | 显示"官方不会索要密码/验证码" |
| 交易密码输入框 | 占位文字: "不要告诉任何人包括客服" |
| 进入安全中心 | 显示本月最新诈骗案例 |
| 群聊固定消息 | 置顶"如何识别假客服"图文教程 |

### 4.8 实施优先级排序

| 优先级 | 方案 | 开发成本 | 防护效果 |
|--------|------|---------|---------|
| **P0** | App 内唯一客服通道 | 低 | 极高 — 根治第三方假客服 |
| **P0** | 敏感操作冷静期 + 勾选声明 | 低 | 高 — 打断冲动操作 |
| **P1** | 屏幕共享/远程控制检测 | 中 | 极高 — 阻断最危险攻击 |
| **P1** | 用户教育融入产品 | 低 | 中 — 长期提升安全意识 |
| **P2** | 官方人员验证系统 | 中 | 中 — 需用户主动使用 |
| **P2** | 社群官方 Bot 验证 | 低 | 中 — 仅覆盖群聊场景 |

## 五、反钓鱼码机制说明

### 5.1 概念

反钓鱼码是用户自定义的秘密标识，平台在所有官方邮件/站内通知中嵌入该标识，用户据此判断通信真伪。

### 5.2 工作原理

```
设置: 用户在安全中心设定自定义文字（如 "我爱吃西瓜2024"）

真邮件:
┌─────────────────────────────────┐
│ 您的提币请求已确认...             │
│ 反钓鱼码: 【我爱吃西瓜2024】      │ ← 能看到 = 真邮件
└─────────────────────────────────┘

假邮件:
┌─────────────────────────────────┐
│ 检测到异常登录，请立即验证...      │
│ （没有反钓鱼码）                  │ ← 没有 = 假邮件
└─────────────────────────────────┘
```

### 5.3 适用性

- 适用于有 **场外通信**（邮件/短信）的场景
- 如果 App 不进行场外联系，传统反钓鱼码作用有限
- 替代方案: **App 内个性化安全标识** — 所有官方弹窗展示用户自设标识，用于辨别真假 App

### 5.4 技术实现

```
数据层:  用户表增加 anti_phishing_code 字段
设置接口: PUT /api/user/security/anti-phishing-code（需 2FA 验证）
邮件发送: 邮件模板中插入用户的 anti_phishing_code
通知推送: 站内信、App Push 中也嵌入该码
```

## 六、建议我们项目补充的安全能力

基于 EBpay 调研，如果我们的 USDT 兑换页面要做安全设计，建议至少覆盖：

1. **活体检测 KYC** — 防止身份伪造
2. **App 内唯一客服通道** — 根治假客服问题
3. **敏感操作冷静期** — 防止社工攻击导致的冲动操作
4. **提币地址白名单** — 限制只向信任地址转账
5. **屏幕共享检测** — 阻断远程控制类诈骗
6. **USDT 地址风控** — 参考 EBpay 的五级评级系统（值得借鉴）
7. **防撞单机制** — 参考 EBpay 的自动地址生成 + 金额浮动（值得借鉴）
