# Agent Team Collaboration（驴压顶 1.0）- PRD（定稿）

## 需求背景
解决 AI 协作中“产出物看似完整，实则逻辑断层”的现状。通过引入工业级颗粒度的“闸门（Gate）”机制，确保 AI 产出的每一步都可审计、可回溯。

## 目标用户画像
- **交付型团队**：拒绝 AI 幻觉，需要 100% 确定性的产品方案。
- **合规/复杂业务 PM**：需要对每一个功能点的变更留痕。

## 功能清单

### P0 - 必须有
| 功能 | 描述 | 验收标准 |
| :--- | :--- | :--- |
| **实时 Linter 拦截** | 锚点校验发生在 AI 写入过程中，而非事后 | P0 冲突直接打断生成，禁止 AI 续写 |
| **强制闸门流转** | 上游未通过审计，下游禁止抓取最新版本 | 系统自动校验 `audit_status` 状态位 |
| **语义修改率双签** | 修改率 > 20% 时触发硬锁定 | 必须由提交 PM 确认说明 + 第二人复核签名解锁 |
| **债务清账闭环** | 例外通行必须关联 Dependency-ID，下一迭代强制置顶 | 不支持二次延期，逾期则封锁该模块提交权限 |

### P1 - 应该有
| 功能 | 描述 | 验收标准 |
| :--- | :--- | :--- |
| **Markdown 语义 Diff** | 标记版本间的逻辑增减而非仅字符差异 | 视觉标注增加、删除、修改条目 |
| **竞品事实关联器** | 调研结论必须超链接至事实证据字段 | 每一条客观结论必须有对应的源数据支撑 |

## MVP 范围
v1.0 仅实现方案 B 定义的“闸门流水线”核心闭环：包含实时 Linter、P0 静态审计、20% 修改率双签、债务化例外通行。

## 验收标准
1. **闭环性**：任一 P0 审计未通过且未办理债务挂账，下游无法获取更新。
2. **唯一性**：双签解锁必须由两个不同身份的用户 ID 完成，严禁自签。
3. **强制性**：债务到期未清，关联模块的后续版本提交功能必须锁定。

## 简化级联分析
- **如果“锚点 Linter”在写入时生效，我们就无需在结束后进行全量的人工对比**。
  - **简化效果**：降低了 40% 的后期审计工作量。
- **如果“Dependency-ID”能强制置顶清账，我们就无需建立独立的“技术债管理系统”**。
  - **简化效果**：流程内自闭环，节省了跨系统的对账成本。

## 保留的分歧
## 分歧点：审计触发频率
选项 A：**全量定期审计** — 优化：一致性 — 适用场景：整体交付。
选项 B：**增量实时审计** — 优化：纠错速度 — 适用场景：敏捷开发，当前采用。
保留策略：异步后台处理，前端展示状态。
触发解决条件：若 v1.0 导致系统明显卡顿，则在 v1.1 增加手动触发选项。
