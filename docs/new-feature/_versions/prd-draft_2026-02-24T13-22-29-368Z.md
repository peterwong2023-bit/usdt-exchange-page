# Agent Team Collaboration（驴压顶 1.0：策划交付流水线）- PRD 草案（PM-Alpha）



> 决策已固化：方案 B 为 V1 唯一实施方案  
> 目标：把“AI 概率产出”变成“可审计、可验收、可冻结、可清账”的工业化策划交付

## 1. 需求背景
多角色并行产出（PRD/Spec/原型/用例/竞品）在真实执行中常“打滑”：
- 漏项：异常态/边界/验收口径缺失，导致测试阶段返工
- 不一致：术语、规则在 PRD/Spec/用例/原型间漂移
- 不可追溯：冻结后改动无证据链，责任与影响范围不清
- AI 失控：长文漂移、幻觉污染、上下文失忆，且容易被“继续流转”掩盖

本功能用**闸门（Gate）+ 锚点（Anchors）+ 反向审计（Audit）+ 债务清账（Dependency-ID）+ 语义锁定双签（>20%）**把交付变成可执行制度。

## 2. 目标用户
- 小型交付团队（PM/UX/QA/Doc）希望“策划包定版后即可开工”
- 需要 AI 提效，但必须保留人工确认与审计链路
- 对质量与合规敏感，希望用制度降低翻车率

## 3. 目标与成功标准
### 3.1 目标（V1）
1) 建立可对账的交付流水线：上游冻结/闭环后才可触发下游
2) 实现三类硬闸门：
   - 锚点 Linter：每次 AI 写入都校验，P0 冲突立即阻断
   - 语义修改率：>20% 进入硬锁定，必须双签强确认
   - 例外通行：必须债务化（Dependency-ID），下一迭代自动置顶清账

### 3.2 成功标准（可验收）
- 任一 P0（Linter 或审计）未闭环：下游无法抓取新版本（硬阻断）
- semantic_rate > 20%：无“双签记录”下游无法抓取（硬阻断）
- 例外通行：无 Dependency-ID 不允许提交；下迭代自动置顶为 P0 清账项
- 交付物齐全且一致：PRD/Spec/用例/原型至少 3 个抽检点可追溯到锚点编号

## 4. 范围与不做
### 4.1 范围（V1）
- 文档流转状态机与闸门控制
- 锚点体系 + Linter（每次 AI 写入触发）
- 反向审计清单与闭环记录
- 语义修改率（规则条目级）+ >20% 双签强确认
- 例外通行 + Dependency-ID 债务墙 + 到期与豁免机制
- Diff（V1）：Markdown 变更标注（非 UI）

### 4.2 不做（V1）
- 自治多 Agent 自组织交付（方案 C 不做）
- UI 级 Diff、复杂可视化审阅器（后续版本）
- 自动写代码/自动发布/自动线上操作（明确禁止）

## 5. 方案对比（已选 B）
### 方案 A：行业标准（人工协作 + 会议推进）
- 优点：组织阻力低
- 风险：漏项/不一致/返工高；不可追溯

### 方案 B：闸门流水线（已选定）
- 核心：闸门阻断 + 锚点契约 + 审计闭环 + 债务清账 + 双签解锁
- 优点：确定性交付，可审计、可验收、可清账
- 风险：需要纪律（但可用硬闸门约束）

### 方案 C：自治多 Agent（不做）
- 风险：合规/可靠性不可控，责任难界定

## 8. 核心工作流（端到端 + 闸门）
### 8.1 主流程
1) PRD 草案（驴来疯）→ 产出 prd-draft.md  
2) Gate-PRD 冻结（驴得稳）→ prd.md（Locked）  
3) Gate-Audit 强制反向审计  
   - 输出 logic-audit.md  
   - 必填 logic-audit-resolution.md（逐条闭环）  
   - 未闭环：阻断进入 Spec/用例  
4) Gate-Anchors 锚点锁定 → context-anchors.md  
5) Gate-Spec 冻结 → spec.md  
6) 并行产出：竞品 / 原型 / QA 用例  
7) Gate-QA：用例闸门（CSV 可导入、核心路径漏项阈值）  
8) 汇编冻结（驴不停蹄）：冻结声明 + 变更记录 + 交付清单

### 8.2 三个“补刀”硬规则（已确认必须落地）
#### 规则 1：语义修改率 >20% 的强确认硬锁定（用户已确认 1.B）
- 触发：semantic_rate > 0.20（规则条目级；无效微改不计入）
- 动作：进入 Hard Stop，禁止下游抓取新版本
- 解锁条件：必须完成 Semantic Override 双签
  - 提交确认：任一 PM（确认“大版本重构”或“疑似跑偏”）
  - 第二人复核：另一 PM 或 PMO（必须不同身份）
  - 记录字段：原因、影响面、需同步更新的交付物列表

#### 规则 2：例外通行必须 Dependency-ID 债务化 + 清账置顶（用户已确认 2.B）
- 例外备案必填：Dependency-ID、Owner、Due-Date、Impact、Approver、Risk-Level
- 自动置顶：下一迭代开始时，所有 OPEN/OVERDUE 的 Dependency-ID 自动置顶为 P0 审计项
- 二次延期：
  - 默认禁止
  - 允许 1 次“豁免性二次延期”，但必须升级审批并记录原因/补救计划/最终截止时间
  - 每个 Dependency-ID 仅允许 1 次豁免

#### 规则 3：锚点校验伴随每次 AI 写入回路的 Linter（用户已确认 3.A）
- 触发：每次 AI 生成/改写 Markdown 后立即执行
- 动作：发现锚点 P0 冲突立即阻断，并输出冲突报告（锚点编号 + 命中片段 + 位置）
- 抽检：验收阶段抽检 3-5 条产出可追溯锚点编号（用于证明锚点不是摆设）

## 9. 权限与审计
- 冻结 PRD/Spec：PM（驴得稳负责最终冻结）
- 语义锁定解锁：双签（提交 PM + 第二人复核）
- 例外通行审批：指定审批人（记录在例外备案中）
- 所有阻断/放行/例外/豁免/双签：必须写入审计日志（可导出）

## 10. 初步验收标准（V1）
1) Linter：每次 AI 写入均有校验记录；任一锚点 P0 冲突必须阻断  
2) 语义锁定：semantic_rate>20% 必须硬锁定；无双签记录禁止下游抓取  
3) 例外：无 Dependency-ID 的例外不允许提交；OPEN/OVERDUE 自动置顶为 P0 清账项；二次延期仅允许 1 次升级审批豁免  
4) 审计闭环：logic-audit 与 resolution 齐全且逐条有结论  
5) 用例：test-cases.csv 可打开且 Steps/Expected 单行序号；核心路径漏项阈值可执行  
6) 一致性抽检：至少 3 个点能从用例/原型/Spec 追溯到锚点编号与规则条目

## 11. 里程碑（建议）
- M1：跑通主流程（冻结→审计闭环→锚点→Spec→用例）  
- M2：接入语义锁定双签 + 例外债务墙 + 到期置顶  
- M3：稳定运行两次迭代，输出复盘（阻断原因 Top10、债务清账率、返工下降幅度）
