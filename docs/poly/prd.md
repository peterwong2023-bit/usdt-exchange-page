## 需求背景

C2C 金融钱包已有资金账户与用户信任基础，但缺少高频资金使用场景。通过集成 Polymarket API 数据源，让用户用钱包内稳定币余额参与预测市场，可提升用户留存与资金周转率。

核心挑战：
- Polymarket 天然链上结算 vs 钱包用户习惯账户余额式体验
- 预测市场监管差异极大，需要可控上线路径
- 链上不确定性（拥堵、失败、滑点）vs 用户对支付级确定性的期待

## 目标用户画像

**主要用户**：
- 钱包存量用户（有稳定币余额、习惯 C2C 交易）：小额资金参与事件预测，追求简单快速可提现
- 高风险偏好用户：追逐热点事件，关注赔率变化与快速成交
- 新手用户：不懂链上操作，希望一键参与

**非目标用户（v1.0-v2.0）**：
- 专业做市商、量化套利机器人
- 跨链资产管理与复杂策略交易用户

## 简化级联分析

**发现的精简机会**：
1. **统一状态机**：订单、持仓、结算使用同一套状态定义，减少状态不一致
2. **费用计算模块化**：convenience fee、链上成本、滑点保护共用一个计费引擎
3. **合规配置中心**：地区、KYC、限额、总闸统一配置，避免分散管理
4. **通用争议流程**：交易失败、结算争议、客诉申诉使用统一工单系统

**架构简化原则**：
- 一个强大的"交易状态机" > 十个分散的状态管理
- 一套通用的"合规配置" > 多个硬编码规则
- 一个统一的"费用引擎" > 各功能独立计费

## 保留的分歧

### 分歧点：双层账本 vs 直路由架构
- **选项 A**：直路由到 Polymarket — 优化：快速上线、低风险 — 适用场景：需求验证阶段
- **选项 B**：双层账本 + 后台对冲 — 优化：用户体验、确定性 — 适用场景：规模化运营阶段
- **保留策略**：阶段化实施，基于数据决策
- **触发解决条件**：链上失败客诉率 > X% 且持续 2 周，或日订单量导致链上成本不可接受

### 分歧点：内容化 vs 交易化产品定位
- **选项 A**：强化内容（AI 摘要、话题流、社交） — 优化：用户粘性、差异化 — 适用场景：内容消费为主
- **选项 B**：专注交易（规则清晰、费用透明、确定性） — 优化：转化率、信任度 — 适用场景：金融工具为主
- **保留策略**：v1.0 专注交易，v2.0 基于留存数据决定内容投入
- **触发解决条件**：7 日留存率低于预期或用户反馈偏向内容需求

## 差异化策略（基于竞品分析）

**vs Polymarket 原生**：
- 优势：钱包内一站式体验，无需学习链上操作
- 差异化：账户余额式交易，避免 Gas 和桥接复杂度

**vs MetaMask Prediction Markets**：
- 优势：更低综合成本（目标 < 4%），更强确定性保障
- 差异化：可降级合规策略，支持模拟盘兜底

**vs Kalshi/传统交易所**：
- 优势：更广泛的事件覆盖（借助 Polymarket 数据）
- 差异化：C2C 用户习惯的产品语言和交互流程

**vs Crypto.com 等**：
- 优势：专注预测市场垂直场景，不做复杂衍生品
- 差异化：渐进式架构演进，可根据需求调整复杂度

## 需求背景

## 保留的分歧

### 分歧点：双层账本 vs 直路由
- **选项A（直路由）**：优化简单性和合规性，适用于验证阶段和小规模场景
- **选项B（双层账本）**：优化用户体验和确定性，适用于大规模和高频场景
- **保留策略**：阶段化实现，基于量化指标决策
- **触发解决条件**：
  - 链上失败导致客诉率>5%且持续2周
  - convenience fee导致转化率下降>20%
  - 日订单量>1000笔导致链上成本不可接受

### 分歧点：功能丰富度 vs 简洁性
- **选项A（功能丰富）**：优化用户留存和差异化，适用于竞争激烈场景
- **选项B（保持简洁）**：优化开发效率和稳定性，适用于MVP验证阶段
- **保留策略**：P0/P1/P2分层，基于用户反馈迭代
- **触发解决条件**：用户留存数据和竞品压力评估

## 竞品差异化策略

**vs Polymarket原生**：
- 优势：钱包内一站式体验，无需学习链上操作
- 差异化：账户式资金管理，确定性更强

**vs MetaMask Prediction**：
- 优势：更低的综合费用（目标<4%），更好的交易保护
- 差异化：渐进式架构演进，可降级到只读模式

**vs Kalshi/传统交易所**：
- 优势：更广泛的事件覆盖，更低的准入门槛
- 差异化：移动优先，社交化元素

**核心差异化**：在监管不确定性下仍能提供稳定服务的"可降级架构"，以及基于钱包生态的资金使用场景创新。

## 1. 需求背景

C2C 金融钱包已有资金账户与用户信任基础，但缺少高频"资金使用场景"。Polymarket 具备强话题性与即时反馈机制，适合拉动高频访问与资金周转。本产品目标是在钱包内提供"类似 Polymarket 的预测市场体验"，通过 Polymarket 公开 API 获取市场数据，让用户用钱包内的稳定币余额（假设为 USDC）进行预测与结算。

**核心矛盾**：
- Polymarket 天然是链上结算；钱包用户更习惯"账户余额式"的下单与结算体验
- 预测市场在不同地区监管差异极大，需要"可控上线路径"

**定稿核心原则**：
- **先验证需求，再升级架构**：阶段2先走直路由+费用覆盖摩擦，阶段3才评估双层账本
- **合规工程化优先**：地区开关、KYC gate、审计流水、交易总闸为上线前置条件，不是约束项
- **确定性优于功能丰富**：宁可功能少，不能让用户在错误价格成交或余额对不上

## 4. 用户故事（按角色分组）

### 作为普通用户（钱包存量用户/新手）

- 我在钱包"预测"频道看到热门事件卡片，能看到标题、到期时间、YES/NO当前价格和参与热度，不需要懂链上知识
- 我点入详情页，能看到事件描述、规则、结算条件、数据来源声明和风险提示，知道"这是Polymarket的数据，不是我们自己的市场"
- 我选择YES，输入10 USDC，系统在下单前告诉我：convenience fee是多少、预估链上成本、报价有效期还剩多少秒、滑点保护参数、预估能买到多少份额
- 我确认后1秒内看到"已受理"，3分钟内看到"已成交"或"未成交已退款"，每个状态都有可追溯的凭证
- 到期后我自动收到结算通知，结算单清晰显示事件结果、我的盈亏和费用明细，钱回到我的USDC余额
- 如果我在受限地区，我看不到预测入口，或者点击时收到明确提示，不会出现"下单成功但后来被撤销"的情况

### 作为高风险偏好用户

- 我能快速找到热门市场，看到实时赔率变化
- 我下单时能设置滑点保护，避免在价格剧烈波动时以远超预期的价格成交
- 我能随时查看持仓估值并选择提前卖出
- 我能在到期前24小时和1小时收到提醒
- 如果系统进入"只平不开"模式，我能清楚看到原因和预计恢复时间

### 作为运营人员

- 我能通过后台配置地区黑白名单，无需发版即可生效
- 我能一键触发交易总闸，暂停全部新单但不影响存量持仓查询
- 我能配置单笔/单日/单市场限额
- 我能导出任意交易的完整链路凭证，用于客服处理和合规审计
- 当市场出现异常时，我能快速下架市场并触发退款流程

### 作为合规/风控人员

- 我能查看完整的KYC状态与地区校验日志
- 我能查看大额交易审核队列并处理可疑交易
- 我能导出指定时间段内的全部交易流水，格式满足监管报送要求
- 我能看到当前系统的风险敞口状态（阶段3引入双层账本后）

## 5. MVP 范围

### v1.0 三阶段路线

```
阶段1（只读验证）→ 阶段2（小额交易直路由）→ 阶段3（评审是否引入双层账本）
```

### 阶段1 — 只读验证兴趣（M1）

**做什么**：
- 市场列表与详情（只读）
- 收藏与分享
- 风险提示与数据源声明
- 地区入口控制（合规P0前置）

**不做什么**：任何涉及资金的功能

**验证目标**：用户是否愿意在钱包里浏览预测市场内容；收藏率、分享率、停留时长

---

### 阶段2 — 小额交易直路由（M2，MVP核心）

**做什么**：
- 全部P0功能（见功能清单）
- 单笔限额（建议初始上限100 USDC，待合规确认）
- 直接路由到Polymarket成交（方案A）
- 费用透明披露 + 滑点保护/IOC + 完整状态机
- 到期自动结算 + 到期提醒

**不做什么**：
- 双层账本/净额对冲（方案B）
- AI摘要、话题流、个性化推荐
- 价格阈值提醒、赔率突破提醒
- 杠杆、借贷、保证金
- 复杂订单类型（限价单、止损单等）

**竞品对标**：与MetaMask Prediction Markets路径一致（钱包入口+直接接Polymarket+明确收费），差异化在于更严格的交易保护和合规工程化

---

### 阶段3 — 评审是否引入双层账本（M3，有门槛）

**触发条件（满足任一）**：
- 链上失败/拥堵导致的交易相关客诉率持续2周超过运营基线（基线在阶段2上线后4周设定）
- Convenience fee提升后交易转化率下降超过20%
- 日订单量达到阈值导致链上成本不可接受（具体阈值待阶段2数据确定）

**若触发**：单独立项，需通过风控/资金/合规专项评审后才可进入开发

**若不触发**：维持方案A并持续优化费用结构与链路稳定性

## 6. 验收标准

### 6.1 系统SLO

| 指标 | 目标值 | 说明 |
|---|---|---|
| 市场列表/详情可用性 | ≥99.5%/月 | 不含计划内维护 |
| 行情延迟 | P95 < 3s | 从我们服务到客户端 |
| 下单受理响应 | P95 < 1s | 返回已受理/已拒绝 |
| 成交回执 | P95 < 180s | 以链上/外部回执为准；超时进入挂起可追踪状态 |
| IOC退款完成 | P95 < 60s | 未成交撤销后退款到账 |
| 结算任务发起 | T+5分钟内 | 市场到期后 |
| 结算完成 | T+60分钟内 | 超时进入争议队列并展示状态 |
| 总闸生效 | ≤30s | 触发后全部新单返回暂停提示 |

### 6.2 业务KPI

| 指标 | 目标值 | 观察周期 |
|---|---|---|
| 交易提交成功率 | ≥98% | 进入"链上提交"状态 |
| 交易最终闭环率 | ≥95% | 成交或未成交撤销且退款完成 |
| 资金差错 | 0笔 | 任何一笔必须可对账找回 |
| 客服首次响应率 | ≥95%/24h | 交易类工单 |
| 合规拦截准确率 | 受限地区拦截率100%；误判率<0.1% | 持续监控 |

### 6.3 阶段2上线后4周业务观察指标

（具体阈值上线前与业务方对齐，以下为观察维度）
- 预测频道渗透率（访问预测频道的活跃用户占比）

## 7. 简化级联分析

### 发现1：状态机统一消除多个特殊处理逻辑

**洞察**：链上失败、部分成交、API断流、市场下架——这些看似不同的异常，本质上都是"订单未能正常闭环"。

**级联简化**：
- 统一用"挂起可追踪"状态承接所有异常
- 统一退款路径：无论何种原因导致未成交，都走"冻结→解冻→退回余额"的标准流程
- 消除了：链上失败专用处理、API断流专用处理、部分成交专用处理——全部收敛到同一状态机

**收益**：减少约3个独立异常处理模块，客服话术统一，用户认知统一

---

### 发现2：IOC + 报价TTL消除"半吊子状态"

**洞察**：CLOB订单可能挂单吃不完，链上可能拥堵导致长时间pending——这是客诉的主要来源。

**级联简化**：
- IOC（未成交即撤销）+ 报价TTL（过期禁止提交）两个机制组合
- 消除了：挂单管理、部分成交处理、用户主动撤单的大部分场景
- 用户只会看到"成交"或"未成交已退款"，不会看到"挂单中"

**收益**：减少约2个持仓管理复杂场景，退款逻辑简化

---

### 发现3：费用透明披露消除事后争议

**洞察**：竞品（MetaMask 4%费用）的主要客诉来源是"我不知道要收这么多费"。

**级联简化**：
- 下单前强制展示所有费用（convenience fee + 预估链上成本 + 滑点区间）
- 消除了：费用争议类客诉、退款申请（因费用不透明导致的）、客服解释成本

**收益**：预计减少30-50%的交易类客诉（参考竞品经验）

```

  优化：降低技术复杂度与运营风险
  适用场景：订单量中等、链上失败率可接受、用户对费用不敏感

```

```

  若差异>15%，选低费率并评估成本覆盖路径
```

## 需求背景

C2C 金融钱包已有资金账户与用户信任基础，但缺少高频"资金使用场景"。Polymarket 具备强话题性与即时反馈机制，适合拉动高频访问与资金周转。

**核心目标**：在钱包内提供预测市场体验，通过 Polymarket 公开 API 获取市场数据，让用户用钱包内的稳定币余额（USDC）进行预测与结算。

**关键矛盾**：
- Polymarket 天然是链上结算；钱包用户习惯"账户余额式"的下单与结算体验
- 预测市场在不同地区监管差异极大，需要可控上线路径
- 链上不确定性（拥堵、失败、滑点）与用户对"支付级确定性"的期望之间存在天然张力

**定稿核心原则**：
1. **先验证需求，再升级架构**：阶段2先走直路由+费用覆盖摩擦，阶段3才评估双层账本
2. **合规工程化优先**：地区开关、KYC gate、审计流水、交易总闸为上线前置条件，不是约束项
3. **确定性体验优先于功能丰富度**：宁可功能少，不能让余额对不上

## 用户故事（按角色分组）

### 作为普通用户（新手/存量钱包用户）...

**发现与理解**
- 我在钱包"预测"频道看到热榜事件卡片（标题、到期时间、YES/NO价格、24h变化、参与热度）
- 点入详情页，看到事件描述（来自Polymarket）、规则、结算条件、数据源声明与风险提示
- 我能清楚看到"这是Polymarket的市场数据，我的订单将路由到Polymarket执行"

**下单流程**
- 我选择YES，输入10 USDC
- 系统展示：预估份额、convenience fee（如1 USDC）、预估链上成本（如0.5 USDC）、报价有效期（30秒倒计时）、滑点保护参数
- 我点击确认：1秒内看到"已受理"；30-180秒内看到"已成交"并附链上凭证，或"未成交已撤销，资金已退回"
- 若报价过期：系统提示"价格已更新，请刷新后重新确认"，不会以旧价格成交

**持仓管理**
- 我在持仓页看到该市场仓位、买入成本、当前估值、可卖出按钮
- 我随时可以卖出（走相同的受理→成交/撤销状态机）
- 若系统触发风控，我看到"当前仅允许卖出，不允许新增买入"的清晰提示

**到期结算**
- 到期后系统自动结算，我收到通知："某事件已结算，盈亏+X USDC，已入账"
- 结算单展示：事件名称、结果、我的仓位、盈亏金额、费用明细
- 若结果有争议，我看到："结算处理中，预计XX小时内完成，如有疑问请联系客服"

### 作为运营人员...

- 我可以通过地区策略中心配置哪些国家/地区可用，变更5分钟内生效
- 我可以一键触发交易总闸，暂停所有新单，存量持仓仍可查看和退出
- 我可以查看实时风控大盘：交易量、成功率、客诉率、链上成本
- 我可以配置单笔/单日限额，以及convenience fee比例
- 当市场出现异常（Polymarket下架/结果争议），我可以将该市场切换为"争议态"并推送公告

### 作为合规/客服人员...

- 我可以导出任意订单的完整链路（报价快照、费用明细、链上tx hash、回执时间戳、裁定来源）
- 我可以查看KYC状态与地区校验记录
- 我可以在5分钟内还原任意一笔资金变动的完整轨迹
- 交易类工单我能在24小时内完成首次响应（系统提供标准化信息面板）

## 简化级联分析

### 发现1：三套"不确定性处理"可以统一为一套状态机

**原问题**：草案中分别描述了"链上失败处理"、"API断流处理"、"市场下架处理"三套独立逻辑，每套都有自己的降级策略。

**级联简化**：所有外部不确定性统一收敛到"禁止新单 + 存量可查可退 + 状态可见"这一个原则。具体触发条件不同，但处理模式相同。

**消除了什么**：不需要为每种异常单独设计UI和运营流程；客服话术可以统一；工程实现只需一套降级框架。

---

### 发现2：方案A的"费用透明+交易保护"可以替代方案B的大部分用户体验目标

**原问题**：方案B的核心价值主张是"让用户感受不到链上不确定性"。但这需要双层账本、净额对冲、风险资金、做市能力。

**级联简化**：通过IOC（未成交即撤销）+ 滑点保护 + 明确的状态机 + 费用透明，用户同样可以获得"确定性体验"——只是确定性的表达方式从"一定成交"变成了"要么成交要么明确撤销退款，绝不悬空"。

**消除了什么**：不需要方案B的全套基础设施就能解决80%的用户体验问题；方案B的剩余价值（降低链上成本、提升吞吐）留到阶段3用数据验证。

---

### 发现3：提醒系统砍到1类，推送基础设施复用钱包现有能力

**原问题**：草案列了价格阈值、到期倒计时、赔率突破等多类提醒，需要实时行情触发引擎。

**级联简化**：MVP只做到期提醒（定时任务，不需要实时触发），复用钱包现有通知系统。

**消除了什么**：不需要新建实时推送基础设施；不需要用户偏好管理；不需要防骚扰策略。节省约1个后端工程师1个月工作量。

---

### 发现4：市场展示不需要内容运营，热榜=成交量排序

**原问题**：草案提到"话题流"、"热搜"、"内容消费"，隐含需要内容团队和推荐算法。

**级联简化**：Polymarket API本身提供成交量、参与人数数据，直接按这两个维度排序即为"热榜"，不需要任何内容运营介入。

**消除了什么**：内容运营岗位依赖、推荐算法、反作弊、冷启动问题全部消失。

## 6. 验收标准（量化 SLO）
1. **交易成功率**：进入链上提交阶段的订单成功率 ≥ 95%。
2. **资金一致性**：账务差错率为 0，所有冻结/扣款必须有对应的成交回执或退款记录。
3. **响应时效**：行情刷新 P95 < 3s；下单受理反馈 < 1s。
4. **合规阻断率**：受限地区用户通过技术手段绕过准入的比例 < 0.1%。

- **洞察**：如果采用 **IOC（Immediate-or-Cancel）订单模式**，我们就不需要实现复杂的“挂单管理”、“撤单队列”和“长时间状态同步”。
- **简化结果**：

  - 降低了客服压力：订单要么成，要么退，没有中间态。

# C2C 钱包内 Polymarket 集成（使用钱包货币预测）- PRD（定稿）

| 版本 | 日期 | 变更摘要 |
|:---|:---|:---|

## 1. 需求背景
C2C 金融钱包拥有大量稳定币存量用户，但缺乏高频资金消耗场景。Polymarket 作为当前最火的预测市场，其数据具有强话题性。本需求旨在通过 API 集成，让用户直接使用钱包内的稳定币（如 USDC）参与预测，提升钱包日活（DAU）与资金周转率。

## 2. 目标用户画像
1. **存量 C2C 用户**：有稳定币余额，不熟悉链上操作（Gas、私钥、桥接），希望一键参与热点事件。
2. **高频投机者**：对赔率敏感，关注事件即时反馈，追求下单的确定性。
3. **合规敏感用户**：位于特定准入地区，需在受控环境下参与。

## 3. 功能清单

### P0 - 必须有（MVP 阶段 - 阶段 2 交易闭环）
| 功能 | 描述 | 验收标准 |
|:---|:---|:---|
| **地区合规开关** | 基于 GeoIP 和 KYC 国籍的准入控制 | 非准入地区用户隐藏入口；持仓用户迁移至非准入区仅允许平仓。 |
| **直路由交易引擎** | 钱包余额扣款，后台代用户在 Polymarket 下单 | 1s 内返回受理状态；3min 内同步链上成交/失败结果。 |
| **交易保护机制** | 引入滑点保护（Max Price）与 IOC（立即成交否则取消） | 实际成交价超过用户确认价的 1% 时自动撤单并退款。 |
| **费用透明披露** | 下单前展示：Convenience Fee + 预估 Gas + 报价有效期 | 用户必须勾选确认费用明细后方可点击下单。 |
| **基础持仓与结算** | 展示 YES/NO 份额、成本、浮动盈亏；到期自动入账 | 市场判定后 60min 内完成资金回滚至钱包余额。 |
| **合规总闸** | 运营后台一键关停交易功能 | 开启后全量用户无法新增开仓，仅保留查询和平仓。 |

### P1 - 应该有（v1.1 迭代）
| 功能 | 描述 | 验收标准 |
|:---|:---|:---|
| **到期提醒** | 市场即将结算前的 Push 通知 | 结算前 2 小时发送提醒，引导用户查看或提前平仓。 |
| **热门分类筛选** | 基于 Polymarket 标签的简单分类（政治、体育、加密） | 首页支持至少 5 个主流分类的快速切换。 |
| **分享卡片** | 生成带赔率的事件分享图 | 支持带邀请码的 H5 分享，点击回流至钱包。 |

### P2 - 锦上添花（看数据决定）
| 功能 | 描述 | 验收标准 |
|:---|:---|:---|
| **双层账本架构** | 钱包内即时撮合，后台批量净额对冲 | 仅在日单量 > 5000 笔或链上失败率 > 10% 时启动开发。 |
| **价格波动提醒** | 赔率剧烈变动推送 | 赔率变化超过 20% 时触发。 |
| **AI 摘要** | 自动生成事件背景介绍 | 仅针对 Top 10 热门市场展示。 |

## 4. 用户故事

### 作为普通用户...
- 我在钱包“发现”频道看到“美国大选”预测，点击进入看到 YES 价格为 0.6 USDC。
- 我输入 100 USDC，系统提示需支付 4% 服务费，我确认后点击买入。
- 页面显示“处理中”，1 分钟后收到“买入成功”通知，持仓增加 160 份 YES。

### 作为运营人员...
- 当收到监管通知或 Polymarket API 出现大规模异常时，我能在后台一键开启“只平不开”模式，保障存量用户利益并阻止新风险。

## 5. MVP 范围
- **做：** 稳定币（USDC/USDT）直接下单、Polymarket 热门列表展示、基础持仓、到期自动结算、地区黑名单、KYC 准入。
- **不做：** 杠杆交易、模拟盘积分、社交聊天室、多链资产支付（v1.0 仅限钱包主币种）。

## 6. 验收标准
1. **资金一致性**：任何失败订单必须在 5 分钟内完成余额解冻，差错率为 0。
2. **性能**：行情刷新 P95 < 3s；下单受理反馈 < 1s。
3. **合规性**：GeoIP 拦截准确率 100%；未实名用户拦截率 100%。
4. **稳定性**：API 异常时必须显示“价格已过期”并禁止下单，严禁使用错误价格成交。

## 7. 简化级联分析
- **如果“IOC（立即成交或取消）”策略成立**：我们就不需要开发复杂的“挂单管理系统”和“部分成交处理逻辑”，大大简化了前端账本状态机。
- **如果“统一使用钱包稳定币结算”成立**：我们就不需要为用户展示链上 Gas 消耗细节（将其封装在服务费中），消除了用户对原生代币（如 MATIC/POL）的认知负担。

## 8. PM 辩论记录
| 议题 | PM-Alpha 观点 | 老驴观点 | 最终决策 | 理由 |
|:---|:---|:---|:---|:---|
| **交易架构** | 首推双层账本，追求即时成交 | 架构太重，先走直路由验证需求 | **阶段化执行** | MVP 追求 ROI，先用 A 方案跑通，达标后再做 B。 |
| **辅助功能** | 增加 AI 摘要和话题流提升转化 | 钱包应聚焦交易，内容太分散精力 | **砍掉/降级** | 减少非核心开发成本，利用 Polymarket 原生描述。 |
| **提醒系统** | 多维度提醒（价格、赔率、到期） | 基础设施重，只做到期提醒 | **仅保留到期提醒** | 稳一稳，先解决“能买能卖”，再解决“好不好用”。 |

## 9. 保留的分歧
```
## 分歧点：手续费模式
选项 A：固定百分比（如 4%） — 优化：商业收益与简单认知 — 适用场景：小额散户
选项 B：动态 Spread（价差） — 优化：风险对冲与大额成交 — 适用场景：高频交易者
保留策略：文档记录，MVP 采用选项 A
触发解决条件：当月交易额突破 $1M 或进入阶段 3（双层账本）时必须切换为 B
```

---
**老驴寄语**：稳一稳，先把合规总闸和资金对账做扎实，别让用户在钱包里丢了钱，这比什么 AI 摘要都强。
