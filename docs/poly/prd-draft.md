## 需求澄清（建议用户确认）
1. 目标用户是谁：现有 C2C 钱包的存量用户（想做“预测/投机”）还是新增的加密原生用户？是否包含商户/做市者/渠道方？
2. “用钱包的货币进行预测”具体指什么：用法币余额下单并以法币结算？还是用钱包内的稳定币/积分/余额做“模拟盘”？涉及杠杆、借贷、抵押吗？
3. 交易与结算的合规边界：是否允许真实资金参与？哪些国家/地区可用？是否必须 KYC/AML？是否需要风控（限额、黑名单、可疑交易）？
4. 与 Polymarket 的关系形态：仅用其公开市场数据做“信息+模拟预测”，还是要把用户的订单真实路由到 Polymarket（链上）并实际成交？若真实成交，谁承担链上 Gas、滑点、失败重试？
5. 成功标准与约束：P0 是提升钱包日活/留存还是创造手续费收入？性能目标（下单时延、行情刷新）、安全要求（资金托管、密钥管理）、以及与现有钱包功能（充值/提现/OTC/C2C）的联动范围是什么？

## 以下基于合理假设的 PRD 草案

# C2C 钱包内 Polymarket 集成（使用钱包货币预测）- PRD 草案（PM-Alpha）v1.0

| 版本 | 日期 | 变更摘要 |
|---|---|---|
| v1.0 | 2026-03-11 | 首版：在 C2C 钱包内以“钱包余额（稳定币）”参与 Polymarket 预测；含行情、下单、结算、风控与合规边界假设。 |

## 1. 需求背景
C2C 金融钱包已有资金账户与用户信任基础，但缺少高频“资金使用场景”。Polymarket 具备强话题性与即时反馈机制，适合拉动高频访问与资金周转。需求目标是在钱包内提供“类似 Polymarket 的预测市场体验”，并通过 Polymarket API 获取市场数据，同时让用户用钱包内的货币（假设为稳定币 USDC 余额）进行预测与结算。

关键矛盾：
- Polymarket 天然是链上结算/特定资产；钱包侧用户更习惯“账户余额式”的下单与结算。
- 合规与风险：预测市场在不同地区监管差异极大，需要“可控上线路径”。

## 2. 目标用户
1) 钱包存量用户（有稳定币余额、习惯 C2C 交易）：希望用小额资金参与事件预测，追求简单、快、可提现的体验。  
2) 高风险偏好用户：追逐热点事件，频繁买卖，关注赔率变化与快速成交。  
3) 新手用户：不懂链上，不想配置钱包/Gas，希望“一键参与”。

非目标（v1.0 不优先）：
- 专业做市、量化、套利机器人。
- 跨链资产管理与复杂策略交易。

## 3. 创新灵感（碰撞区思维 + 反转练习的发现）
- 碰撞区：把“预测市场”当成“钱包里的余额理财/竞猜额度”。用户不关心链上订单簿，只关心“我投了多少、现在值多少、结果如何、可以提现吗”。因此可用“账户式仓位 + 统一结算”重构体验。
- 反转练习：
  - 常规：用户下单=链上交易。反转：用户下单=钱包内部记账，链上只是后台对冲/路由（或先模拟后实盘分阶段）。
  - 常规：先选择市场再下注。反转：从用户关心的“话题流/热搜”进入，下注只是信息消费的自然延伸。
- 极端尺度启发：如果行情 API 不稳定/链上拥堵，仍要保证钱包余额不乱、用户能看到明确状态（“已受理/已成交/失败回滚”），因此交易状态机必须像支付一样严谨。

## 4. 方案对比

### 方案 A：标准集成（信息 + 路由下单到 Polymarket）
- 核心功能点
  - 拉取 Polymarket 市场列表/价格/成交量/到期时间（API）。
  - 钱包内展示市场详情、YES/NO 两边价格。
  - 用户用钱包稳定币余额下单；后台将稳定币兑换/桥接为 Polymarket 所需资产并在链上下单（代用户执行）。
  - 成交后在钱包内显示持仓、浮动盈亏；到期结算后自动入账。
- 用户故事
  - 用户看到“某事件 YES 0.62”，输入 10 USDC 买入 YES，几秒后显示“已成交”，到期后自动结算为 USDC。
- 创新亮点
  - “免 Gas、免链上学习成本”的托管式交易体验。
- 潜在风险
  - 合规风险高（实盘预测市场）。
  - 链上失败、滑点、部分成交导致用户体验不稳定。
  - 资金托管/代理交易带来牌照与风控要求。
- 极端尺度测试结果
  - 10 万用户同时下单：链上吞吐与重试风暴，状态一致性压力巨大。
  - API 断流 1 小时：无法报价/成交回执，需降级为“暂停下单，仅展示缓存价格+风险提示”。
  - 市场异常/被下架：需强制平仓/退款策略，否则资金纠纷。

### 方案 B：创新优化（双层账本：钱包内“即时报价成交” + 后台批量对冲）
- 核心功能点
  - 前台对用户提供“钱包内撮合/报价成交”（以 Polymarket 价格为基准价 + 风险缓冲）。
  - 后台按时间窗（如每 30 秒/1 分钟）汇总净敞口，批量路由到 Polymarket 对冲（或与外部流动性方对冲）。
  - 用户侧订单始终“账户式成交”，减少链上不确定性。
  - 引入“风险缓冲价差/最大成交额/暂停开仓”机制控制敞口。
- 用户故事
  - 用户点击买入即刻成交并生成仓位；后台稍后完成对冲，用户无感知；若对冲失败，进入“风险模式”并停止新增开仓。
- 创新亮点
  - 把“链上不确定性”从用户链路剥离，体验接近支付/交易所。
  - 通过净额对冲降低手续费与链上拥堵影响。
- 潜在风险
  - 钱包方承担短期市场风险（对冲前价格波动）。
  - 需要更强的风控与定价模型（spread、限额、熔断）。
  - 合规依旧偏高（实盘），但比方案 A 交易一致性更可控。
- 极端尺度测试结果
  - 行情剧烈波动（1 分钟 ±20%）：需动态扩大价差、降低限额、触发熔断，否则对冲损失扩大。
  - 对冲通道故障 2 小时：系统自动切“只平不开/仅允许减仓”，保障敞口不继续扩大。
  - 1 亿笔小额订单：前台账本与风控计算需高性能（内存撮合/风控），但链上只产生少量净额交易，整体可扩展性更强。

### 方案 C：颠覆性（先“预测积分/余额凭证”，后“可选实盘映射”）
- 核心功能点
  - v1.0 上线为“信息 + 钱包内预测积分赛道”：用户用钱包余额购买“预测凭证”（非对外可转让），结算按规则派奖；市场数据来自 Polymarket，但不直接路由其订单。
  - 通过“分级解锁”实现合规与风控：完成 KYC、地区允许、风险测评后，才可切换到“实盘模式”（未来版本）。
  - 增加“话题流 + AI 摘要”：每个市场自动生成事件解释、关键变量与风险提示，降低认知门槛。
- 用户故事
  - 用户在钱包热榜看到事件，花 5 USDC 参与预测；到期后按结果分配奖励；全程无需链上成交确认。
- 创新亮点
  - “把预测市场当成内容消费与互动工具”，先做高频、低摩擦、低外部依赖的 MVP。
  - 可在合规不确定下快速上线验证需求：留存、下注转化、热门品类。
- 潜在风险
  - 与用户原始期待（“就是 Polymarket”）存在落差，需明确标识“数据源 vs 交易场所”。
  - 奖励/结算规则设计若不透明易引发争议。
- 极端尺度测试结果
  - Polymarket 市场数据停止：可切换到备用数据源或冻结新场次；历史凭证按既定规则结算，不依赖链上。
  - 监管收紧：可直接下线“真钱参与”，保留“内容+模拟”模式，业务连续性更强。
  - 用户量暴增：纯内盘记账扩展性强，但需防刷与反作弊。

## 5. 推荐方案及理由
推荐方案 B（双层账本：前台账户式成交 + 后台批量对冲），分阶段落地：
- 阶段 1（小流量灰度）：引入严格限额与熔断，验证“预测”对钱包留存与资金使用率的提升。
- 阶段 2（扩大品类）：完善风控与定价，逐步降低价差、提高深度。
理由：
- 用户体验最接近“钱包内一键交易”，不被链上失败打断；
- 技术与运营可控：链上不确定性被后台吸收，通过净额对冲降低成本；
- 相比方案 A，一致性与可扩展性更强；相比方案 C，更贴近“用钱包货币真实预测”的产品承诺。

## 6. 用户故事（推荐方案详细版）
1) 发现与理解
- 作为钱包用户，我在“预测”频道看到热榜事件卡片（标题、到期时间、YES/NO 价格、24h 变化、参与人数）。
- 点入详情页，看到事件解释、规则、数据源（Polymarket API）、结算条件与风险提示。

2) 下单与成交
- 我选择 YES，输入金额（默认用钱包 USDC 余额），看到预估可获得份额、当前价格、价差提示、最大可买额度。
- 点击确认后，订单立刻显示“已成交”，持仓里出现该市场仓位、成本与实时估值。

3) 持仓管理
- 我可以随时卖出（减仓/平仓），也可以设置提醒（价格波动、到期前 X 小时、赔率突破阈值）。
- 若系统进入“风险模式”（对冲通道异常/波动过大），页面清晰提示“仅允许卖出，不允许新增买入”。

4) 到期结算与资金回到钱包
- 到期后系统自动拉取结果并结算，生成“结算单”（事件、结果、盈亏、手续费/价差成本）。
- 盈亏以 USDC 形式回到钱包余额，可用于 C2C、转账或提现（取决于钱包现有能力）。

## 7. 初步验收标准（MVP）
### 7.1 功能验收
1. 市场列表
- 可展示市场标题、到期时间、YES/NO 价格、成交热度；刷新间隔可配置（默认 30s，最低 5s，最高 5min）。
2. 市场详情
- 展示规则、结算条件、风险提示、数据源声明；支持买入/卖出。
3. 交易与账本
- 下单后 1 秒内生成“已成交/已拒绝”结果（账户式成交）。
- 订单/持仓/账变三者一致：任意时刻可通过交易流水追溯余额变化。
4. 结算
- 到期后在 5 分钟内完成结算入账（以数据源返回/内部裁定为准，需有异常处理流程）。
5. 提醒与风控
- 支持至少 2 类提醒：价格阈值、到期倒计时。
- 支持限额：单笔、单市场、单日；支持“只平不开”开关。

### 7.2 性能与稳定性
- 行情接口失败时：降级为“显示缓存 + 禁止下单 + 明确提示”，不出现错误价格下单。
- 订单状态机：至少包含 已受理/已成交/已拒绝/已撤销/结算中/已结算/异常挂起。

### 7.3 合规与安全（按假设的最低要求）
- KYC：参与预测需完成基础 KYC（可配置灰度）。
- 地区限制：支持按国家/地区开关；命中限制直接隐藏入口或禁止下单。
- 资金安全：钱包余额冻结/解冻逻辑可追溯；关键操作需要二次确认（支付密码/生物识别/OTP 取其一）。

## 8. 约束与依赖（基于假设）
- 依赖 Polymarket API：市场列表、价格、到期、结果（若无官方结果 API，需要自建结果裁定来源与争议机制）。
- 依赖钱包现有模块：余额账户、冻结、交易流水、KYC、风控限额、通知系统。
- 运营依赖：市场上架/下架策略、黑天鹅事件应急预案（暂停交易、仅平仓、延迟结算）。

## 9. 里程碑（建议）
- M0：合规模型与地区策略确定（能否实盘、哪些地区可用）
- M1：只读行情 + 收藏 + 提醒（验证兴趣）
- M2：账户式买卖 + 小额限额 + 灰度对冲
- M3：扩大市场品类 + 完整结算单 + 风险模式闭环
