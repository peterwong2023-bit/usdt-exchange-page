# 冷钱包模块 — 操作流程文档

> 版本：v1.0 | 更新：2026-03-20
> 用途：运维操作参考、开发实现参考、新人培训

---

## 总览

冷钱包模块包含三条核心操作主线：

| 主线 | 方向 | 运维介入 | 触发方式 |
|------|------|---------|---------|
| 主线一：地址准备 | — | 全程参与 | 运维主动操作 |
| 主线二：冷归集 | 热池 → 冷钱包 | 不需要，全自动 | 余额 > HOT_MAX |
| 主线三：热回笼 | 冷钱包 → 业务地址 → 热池 | 深度参与 | 余额 < HOT_MIN |

---

## 主线一：冷钱包地址准备（从无到可用）

### 前置准备

在地址生成之前，需要先完成地址池的创建和商户绑定。

#### A. 创建地址池

| 属性 | 说明 |
|------|------|
| 操作人 | 管理员 |
| 操作位置 | VOSS 后台 → 冷钱包 → 地址池管理 → "创建地址池"按钮 |
| 填写内容 | 池名称（如"华东入池-A"）、备注等 |
| 产出 | 一个空的地址池（0 个地址） |

#### B. 商户绑定地址池

| 属性 | 说明 |
|------|------|
| 操作人 | 管理员 |
| 操作位置 | VOSS 后台 → 地址池管理（或商户配置页面） |
| 规则 | 一个池可以绑定多个商户 |
| 意义 | 决定冷归集时"该商户的资金往哪些冷地址归"；如需换一批冷地址，直接换绑池子即可 |

> 准备 A、B 完成后，池子才能接收地址。

---

### Step 1：生成地址

| 属性 | 说明 |
|------|------|
| 操作人 | 运维 |
| 操作位置 | Smart App 连接 Ledger 硬件签核器 |
| 操作内容 | Ledger 按 HD 路径（如 m/44'/195'/0'/0/0）派生一批 TRON 地址 |
| 产出 | 一批地址文件（按批次管理，如"批次 #8"） |

### Step 2：地址自动推送到地址库

| 属性 | 说明 |
|------|------|
| 操作人 | 系统自动 |
| 说明 | Smart App 创建完成后，地址自动推送到 VOSS 后台地址库 |
| 运维无需操作 | 不需要手动导入 |
| 状态变化 | 地址出现在地址库列表，验证状态 = `未验证` |

> Step 1 和 Step 2 是连续动作：运维在 Smart App 操作完，回到 VOSS 后台即可看到新地址。

### Step 3：发起验证

| 属性 | 说明 |
|------|------|
| 操作人 | 管理员 |
| 操作位置 | VOSS 后台 → 地址库 → 勾选地址 → 点"验证"按钮 |
| 系统动作 | 自动创建一条设备任务（类型 = 地址验证） |
| 状态变化 | `未验证` → `验证中` |

### Step 4：Ledger 设备验证

| 属性 | 说明 |
|------|------|
| 操作人 | Smart App 自动领取任务 → Ledger 核对地址 |
| 本质 | Ledger 确认"这个地址确实是我派生的"，防止中间人篡改 |
| 状态变化 | 设备任务 `claimed` → `executing` → `completed`；地址 `验证中` → `已验证` |

### Step 5：分配到地址池

| 属性 | 说明 |
|------|------|
| 操作人 | 管理员 |
| 操作位置 | VOSS 后台 → 地址库 → 勾选地址 → "调整地址池"按钮 |
| 操作内容 | 选择目标池（如"华东入池-A"），将地址分配进去 |
| 前提 | 地址已验证 + 目标池已创建且已绑定商户 |
| 状态变化 | poolId: `null` → 对应池 ID |

### Step 6：激活地址

| 属性 | 说明 |
|------|------|
| 操作人 | 管理员点"激活" → 系统执行 |
| 操作位置 | VOSS 后台 → 地址库 → 操作栏"激活" |
| 操作内容 | 从该地址绑定商户的**激活池**发一笔小额 TRX 到该地址 |
| 目的 | 地址有 TRX 后才能在链上接收和处理 USDT 转账 |
| 状态变化 | isActivated: `false` → `true` |

### Step 7：同步余额

| 属性 | 说明 |
|------|------|
| 操作人 | 系统自动 / 管理员手动点"同步余额" |
| 操作内容 | 查询链上该地址的 TRX + USDT 余额 |
| 目的 | 确认激活成功（TRX 到账）+ 记录初始余额 |

### Step 8：地址可用

此时该地址可以作为冷归集的目标地址。

---

### 主线一完整状态流转

```
【前置】创建地址池 → 绑定商户
                          ↓
Smart App + Ledger 生成地址
        ↓（系统自动推送）
地址出现在 VOSS 地址库（未验证）
        ↓
管理员点"验证"（验证中）
        ↓
Ledger 签核通过（已验证）
        ↓
管理员点"调整地址池"分配到池子
        ↓
管理员点"激活" → 激活池发 TRX（已激活）
        ↓
同步余额 → 确认就绪
        ↓
地址可用
```

## 主线二：冷归集（热池 → 冷钱包）

### 前提

商户有已绑定的地址池，池内有可用的冷钱包地址（主线一已完成）。

### Step 1：触发

| 属性 | 说明 |
|------|------|
| 触发条件 | 热池余额 > HOT_MAX |
| 触发方式 | 线上系统自动检测 / 测试环境管理员手动创建 |
| 防重复 | 同商户已有活跃冷归集任务时，不再创建新任务 |

### Step 2：系统选定冷钱包地址 + 计算金额

| 属性 | 说明 |
|------|------|
| 选择策略 | 从商户绑定的地址池中，选**余额最少**的冷钱包地址 |
| 归集金额 | 余额 − HOT_TARGET（精确金额，以 HOT_TARGET 为锚点） |

### Step 3：任务创建 + 自动拼单

| 属性 | 说明 |
|------|------|
| 任务状态 | `待处理` |
| 拼单逻辑 | 系统自动将多个热池地址的归集合并为一笔任务 |
| 展示 | 任务列表显示为一条任务；详情内展示拼单笔数和各源地址明细 |

### Step 4：系统全自动执行

| 属性 | 说明 |
|------|------|
| 操作人 | 系统（**全自动，无需运维介入，无需 Ledger 签核**） |
| 操作内容 | 直接执行链上转账（多个热池地址 → 冷钱包地址） |
| 状态变化 | `待处理` → `处理中` → `已发起`（链上广播）→ `已确认`（链上确认/上链）→ `已完成` |

### Step 5：完成

| 属性 | 说明 |
|------|------|
| 结果 | 资金到达冷钱包地址 |
| xbb 推送 | **无**（冷归集成功不发通知） |

---

### 主线二完整流转

```
热池余额 > HOT_MAX
        ↓
系统创建冷归集任务
（选定余额最少的冷地址 + 计算归集金额 + 自动拼单）
        ↓
  待处理 → 处理中（全自动，无需运维操作）
        ↓
  已发起（链上广播）
        ↓
  已确认（链上确认）
        ↓
  已完成
```

## 主线三：热回笼（冷钱包 → 业务地址 → 热池）

### 前提

商户有可用的冷钱包地址且有余额。

### Step 1：触发

| 属性 | 说明 |
|------|------|
| 触发条件 | 热池余额 < HOT_MIN |
| 触发方式 | 线上系统自动检测 / 测试环境管理员手动创建 |
| 防重复 | 同商户已有活跃热回笼任务时，不再创建新任务 |
| 回笼金额 | HOT_TARGET − 当前余额（精确金额） |
| 系统指定 | 目标业务地址（商户现有的业务地址） |
| 系统不指定 | 来源冷钱包（运维自行决定从哪个冷钱包转出） |

### Step 2：任务创建

| 属性 | 说明 |
|------|------|
| 任务状态 | `待处理` |

### Step 3：派发设备任务

| 属性 | 说明 |
|------|------|
| 操作人 | 系统 |
| 操作内容 | 创建设备任务（类型 = 热回笼），派发到 Smart App |
| 状态变化 | `待处理` → `处理中` |

### Step 4：运维在 Smart App 确认

| 属性 | 说明 |
|------|------|
| 操作人 | 运维 |
| 操作内容 | 在 Smart App 看到热回笼任务，确认回笼金额和目标业务地址 |

### Step 5：运维在 VOSS 后台委托能量

| 属性 | 说明 |
|------|------|
| 操作人 | 运维（手动操作） |
| 操作位置 | VOSS 后台 |
| 操作内容 | 用商户的能量池，将 Energy 委托给自己选定的来源冷钱包地址 |
| 目的 | 冷钱包地址有 Energy 才能发起 USDT 转账，否则要烧 TRX |
| 时机 | 每次转账前按需委托 |

### Step 6：运维手动转账

| 属性 | 说明 |
|------|------|
| 操作人 | 运维 |
| 操作位置 | 外部钱包（不在 VOSS 系统内） |
| 操作内容 | 从选定的冷钱包转账到系统指定的业务地址 |
| 状态变化 | 系统扫块后自动变为 `已发起` |

### Step 7：系统检测到账

| 属性 | 说明 |
|------|------|
| 操作人 | 系统（区块扫描） |
| 操作内容 | 扫块检测到目标业务地址收到入款 |
| 完成判定 | 扫到业务地址收到金额即标记完成 |
| 状态变化 | `已发起` → `已确认` → `已完成` |

### Step 8：资金自动归集到热池

| 属性 | 说明 |
|------|------|
| 操作人 | 系统 |
| 操作内容 | 业务地址余额达到归集阈值后，自动走常规归集流程到热池 |
| 归属 | 属常规归集流程，不再归属热回笼任务 |

---

### 主线三完整流转

```
热池余额 < HOT_MIN
        ↓
系统创建热回笼任务
（计算金额 = HOT_TARGET − 余额，指定业务地址，不指定来源冷钱包）
        ↓
  待处理 → 派发设备任务到 Smart App → 处理中
        ↓
运维在 Smart App 确认任务
        ↓
运维在 VOSS 后台委托能量给冷钱包地址
        ↓
运维用外部钱包：冷钱包 → 业务地址
        ↓
  系统扫块 → 已发起 → 已确认 → 已完成
        ↓
业务地址资金自动归集到热池（常规流程）
```

## 三条主线对比

| | 主线一（地址准备） | 主线二（冷归集） | 主线三（热回笼） |
|---|---|---|---|
| **目的** | 让冷钱包地址可用 | 热池超限资金转到冷钱包 | 热池不足从冷钱包补回 |
| **触发** | 运维主动操作 | 余额 > HOT_MAX 自动触发 | 余额 < HOT_MIN 自动触发 |
| **运维介入** | 全程参与（生成、验证、分配、激活） | **不需要**，全自动 | **深度参与**（确认、委托能量、手动转账） |
| **Ledger** | 生成地址 + 验证签核 | 不需要 | 不需要 |
| **能量委托** | 不需要 | 不需要 | 需要（转账前手动委托） |
| **拼单** | 不涉及 | 有（多个热池地址合并一笔） | 无 |
| **xbb 推送** | 无 | 无 | 有（余额不足告警等） |
| **目标金额** | — | 余额 − HOT_TARGET | HOT_TARGET − 余额 |
| **状态流转** | 未验证→已验证→入池→已激活→可用 | 待处理→处理中→已发起→已确认→已完成 | 待处理→处理中→已发起→已确认→已完成 |

## 附录：后台页面与主线的关系

| 后台页面 | 作用 | 涉及主线 |
|---------|------|---------|
| 地址库 | 管理冷钱包地址的全生命周期 | 主线一 |
| 地址池管理 | 创建/管理地址池、绑定商户 | 主线一（前置） |
| 设备任务 | 查看 Ledger 签核任务状态 | 主线一（验证）、主线三（热回笼） |
| 调度中心 | 监控水位 + 管理冷归集/热回笼任务 | 主线二、主线三 |
| 操作记录 | 全局日志，追溯所有操作 | 全部主线 |
