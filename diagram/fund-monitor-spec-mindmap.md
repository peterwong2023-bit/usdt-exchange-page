## 1. 页面基本信息

- **页面名称**：资金监控中心
- **浏览器标题**：资金监控中心（TRON-USDT）- VVOSS Admin v3
- **所属模块**：资金与交易
- **面包屑**：资金与交易 > 资金监控中心
- **目标用户**：运营 / 财务
- **核心职责**：监控商户热池资金水位，管理自动回储（热池 → 冷钱包）流程
- **架构前提**：每个商户拥有独立热池，不存在"全部商户"聚合视图
- **数据刷新**：页面加载时拉取一次，手动刷新浏览器更新（无自动轮询、无 WebSocket）
- **链/币种**：当前仅支持 TRON-USDT（多链场景后续迭代）

## 4. 水位线模块（实时数据）

### 4.1 配置参数（每个商户独立配置，商户编辑页设置）

| 参数 | 含义 | 本页面是否可改 |
|------|------|--------------|
| HOT_MIN | 最低线，低于此值告警需补给 | 否（只读展示） |
| HOT_TARGET | 目标线 = (HOT_MIN + HOT_MAX) / 2 | 否（自动计算） |
| HOT_MAX | 最高线，超过此值触发回储 | 否（只读展示） |
| BATCH_MIN | 最低回储金额，单次回储的最低门槛 | 否（只读展示） |
| ENERGY_THRESHOLD | 能量阈值，执行回储需要的最低能量 | 否（只读展示） |
| ENERGY_PER_TX | 每笔交易能量消耗（TRC20 Transfer） | 否 |
| ENERGY_SAFETY | 能量安全系数（默认 1.2） | 否 |

#### 推荐值功能（后续迭代）

本页面将提供 HOT_MIN 和 HOT_MAX 的推荐值，辅助运营配置：

**HOT_MIN 推荐公式**：
```
HOT_MIN = PeakOut_per_hour × H_min × Safety
```
- `PeakOut_per_hour`：过去 14~30 天的每小时出款峰值（取 P95）
- `H_min`：最小扛住时长，即即使完全无入款也能扛住的小时数（常用 6h）
- `Safety`：保险系数（常用 1.1 ~ 1.3）
- 直觉：能扛住 H_min 小时出款尖峰的钱

**HOT_MAX 推荐公式**：
```
HOT_MAX = PeakOut_per_hour × H_max × Safety
```
- `PeakOut_per_hour`：同最低线，过去 14~30 天的每小时出款峰值（取 P95）
- `H_max`：最大扛住时长（常用 24h），超过此时长的资金不应留在热池
- `Safety`：保险系数（同最低线）
- 直觉：最低线和最高线本质逻辑一致（峰值 × 时长 × 系数），区别仅在于"扛多久"——最低线保最小安全，最高线封最大上限

**HOT_TARGET**：自动取 (HOT_MIN + HOT_MAX) / 2

### 4.2 水位条（可视化）

| 区间 | 范围 | 颜色 | 含义 |
|------|------|------|------|
| 危险区 | 0 ~ HOT_MIN | 红色 | 余额不足，需补给 |
| 正常区 | HOT_MIN ~ HOT_TARGET | 绿色 | 水位健康 |
| 预警区 | HOT_TARGET ~ HOT_MAX | 橙色 | 可能需要回储 |
| 超限区 | HOT_MAX ~ ∞ | 深橙/红 | 超上限，应立即回储 |

- 三条标记线：最低线 / 目标线 / 最高线（各带数值标注）
- 当前余额圆点（颜色随所在区间变化）

### 4.3 智能状态条

描述的是"**新任务创建**"的条件，不影响已存在的任务。

#### 状态机（6 种状态）

| 状态 | 触发条件 | 样式 | 主数字 | hint |
|------|---------|------|--------|------|
| 余额不足 | HOT_BALANCE < HOT_MIN | 红色 | 低于最低线 {差额} USDT | 请尽快从冷钱包补给，避免影响出款 |
| 无需回储 | HOT_MIN ≤ HOT_BALANCE ≤ HOT_TARGET | 绿色 | （不显示） | 热池水位正常，不会创建新的回储任务 |
| 未触发新回储 | HOT_BALANCE > HOT_TARGET 且 可回储 < BATCH_MIN | 橙色 | 还差 {缺口} USDT | 达到最低回储金额后系统将自动创建并执行新任务（已有 X 条任务不受影响） |
| 新回储待执行 | 可回储 ≥ BATCH_MIN 且能量不足 | 橙色 | 等待能量满足 | 金额已达标，条件满足后将立即执行（已有 X 条任务不受影响） |
| 新回储已触发 | 可回储 ≥ BATCH_MIN 且能量满足 | 绿色 | 条件满足，已自动执行 | 系统已从归集地址拼单回储至冷钱包 |
| 超出上限 | HOT_BALANCE > HOT_MAX | 红色 | 超上限 {超出额} USDT | 超过最高线，系统已自动触发回储 |

#### 状态优先级

```
超出上限 > 余额不足 > 其余按条件顺序判断
```

#### 已有任务提示

当存在活跃任务时，hint 末尾追加 `（已有 X 条任务不受影响）`，避免运营误解为已有任务被取消。

### 4.4 条件 Pills（2 个，横排）

> 注意：已移除"闲置"条件，只保留金额和能量。

| 条件 | 满足时 | 未满足时 |
|------|--------|---------|
| 金额 | 绿色：`金额 {可回储}/{BATCH_MIN} ✓` | 红色：`金额 差{缺口}` |
| 能量 | 绿色：`能量 可转{tx_cap}笔 ✓` | 橙色：`能量 差{缺口}笔` + `→ 阻塞{N}条任务`（如有） |

**能量计算口径**：
```
tx_cap  = floor(当前能量 / (ENERGY_PER_TX × ENERGY_SAFETY))
tx_need = ceil(ENERGY_THRESHOLD / (ENERGY_PER_TX × ENERGY_SAFETY))
```

- 主展示用"笔数"（运营友好），原始能量数值仅放 tooltip
- 能量不足时运营无需操作，系统自动等待能量恢复后执行

## 5. 回储任务列表（实时数据）

### 5.1 任务生命周期

```
系统扫描 → 发现满足条件 → 自动创建任务 → 等待能量 → 能量满足后立即执行 → 成功/失败
```

- **创建时机**：系统定时扫描发现余额超目标线且达到 BATCH_MIN 时自动创建
- **执行时机**：条件满足后**立即执行**（不等待下次扫描）
- **核心原则**：水位线状态只影响"新任务创建"，不影响已有任务执行

### 5.2 任务类型

**统一为一种类型**，不再区分"普通"和"强制"。

- 回储金额 = HOT_BALANCE − HOT_TARGET（余额超出目标线的部分）
- 超过 HOT_MAX 时的差异仅用**告警级别**体现（页面红色 + Telegram 紧急通知）

### 5.3 任务状态机（4 种状态）

| 状态 | 含义 | tag 颜色 | 备注列显示 |
|------|------|---------|-----------|
| 等待能量 | 任务已创建，能量不足 | 黄色/橙色 | 阻塞原因（如"能量等待 可转 1 笔 / 需 2 笔"） |
| 回储中 | 正在执行（含拼单组装+链上确认） | 蓝色 | 执行策略（如"拼单回储 · 5个地址 · 已轮换优先"） |
| 已完成 | 链上确认成功 | 灰色 | — |
| 失败 | 执行失败 | 红色 | 失败原因 + "重试"按钮 |

```
          ┌───────────┐
          │  等待能量   │
          └─────┬─────┘
                │ 能量满足（立即执行）
          ┌─────▼─────┐
          │   回储中    │
          └──┬─────┬──┘
             │     │
      ┌──────▼┐  ┌─▼──────┐
      │ 已完成 │  │  失败   │
      └───────┘  └────┬───┘
                      │ 手动重试
                ┌─────▼─────┐
                │   回储中    │
                └───────────┘
```

### 5.4 失败重试机制

- **不自动重试**
- 失败后 Telegram 通知运营
- 运营在页面点"重试"按钮，任务状态：失败 → 回储中
- 失败原因在备注列展示

### 5.5 回储执行规则

| 规则 | 说明 |
|------|------|
| 执行方式 | 拼单回储（从多个归集地址汇集到冷钱包） |
| 地址优先级 1 | 已轮换过的归集地址优先 |
| 地址优先级 2 | 钱包余额大额优先 |
| 地址上限 | 不在本页面关心，后端配置决定 |

### 5.6 列表结构

#### 摘要行

- 活跃任务数 + 状态分布 tags（等待能量 ×N、回储中 ×M）
- 等待能量任务的阻塞原因汇总（如"N 条因能量不足等待中"）
- 合计金额

#### 活跃任务表

| 列 | 说明 |
|----|------|
| 单号 | 回储单号 |
| 金额（USDT） | 回储金额 |
| 状态 | 等待能量 / 回储中 / 失败 |
| 备注 | 阻塞原因或执行策略 |
| 交易哈希 | 链上 TxID |
| 创建时间 | 任务创建时间 |
| 完成时间 | 任务完成时间（未完成显示"—"） |

> 注意：已移除"类型"列（不再区分普通/强制）

#### 已完成任务（折叠）

- 默认折叠，点击展开
- 显示条数 + 合计金额
- 列：单号 / 金额 / 状态 / 交易哈希 / 创建时间 / 完成时间

### 5.7 手动触发扫描

| 属性 | 说明 |
|------|------|
| 位置 | 回储任务列表标题栏右侧 |

## 9. 告警体系

### 9.1 页面内告警

| 触发条件 | 展示位置 | 展示方式 |
|---------|---------|---------|
| HOT_BALANCE < HOT_MIN | KPI"热池余额" + "距目标线" | 红色数字 + 红色提示文案 |
| HOT_BALANCE < HOT_MIN | 水位线状态条 | 红色"余额不足"状态 |
| HOT_BALANCE > HOT_MAX | 水位线状态条 | 红色"超出上限"状态 |
| 有等待能量任务 | 能量 pill | 橙色 + "阻塞 N 条任务" |
| 有失败任务 | 回储任务列表 | 红色 tag + 重试按钮 |
| 出款有异常 | Drawer 出款明细 | 黄色提示条 |

### 9.2 Telegram 告警（后续细化）

| 场景 | 说明 |
|------|------|
| 低于 HOT_MIN | 通知运营从冷钱包补给 |
| 超过 HOT_MAX | 紧急通知（告警级别更高） |
| 回储任务失败 | 通知运营手动处理 |
| 告警频率 | 待定（每次扫描报 vs 只报一次） |
| 恢复通知 | 待定（余额回到 Min 以上后是否通知"已恢复"） |

## 11. 已移除/简化项

| 项目 | 原状态 | 变更 | 原因 |
|------|--------|------|------|
| "全部商户" | 曾有下拉选项 | 删除 | 每个商户独立热池，聚合无意义 |
| 闲置条件 | 三条件之一 | 删除 | 本意是空闲时执行，已被能量安全系数覆盖 |
| 强制/普通类型 | 两种类型 | 合并为一种 | 达到 MAX 时可回储必然 ≥ BATCH_MIN，无实质差异 |
| READY vs EXECUTING | 两个状态 | 合并为"回储中" | 运营不关心内部阶段，"回储中"更友好 |
| 等待执行 | 原状态名 | 改为"等待能量" | 去掉闲置后，等待的只有能量 |
| 强制回储中 | 水位线状态 | 改为"超出上限" | 去掉强制概念，用告警级别表达 |
| 导出功能 | 有导出按钮 | 删除 | — |
| 归集健康面板 | 曾有独立面板 | 删除 | 实用性低 |
| 口径说明 Tab | Drawer 第三个 Tab | 删除 | 改为字段旁 tooltip |
| 期初/期末余额 | Drawer 汇总 | 删除 | — |
