## 1. 页面基本信息

- **页面名称**：调度中心
- **浏览器标题**：调度中心（TRON-USDT）- VVOSS Admin v3
- **所属模块**：冷钱包 > 调度中心（取代原资金调度页面）
- **定位**：资金调度的高级看板版本——在现有资金调度任务管理基础上叠加可视化监控和决策辅助层
- **目标用户**：运维
- **核心职责**：
  1. 监控商户热池资金水位，可视化趋势和 KPI
  2. 管理冷归集（热池 → 冷钱包）和热回笼（冷钱包 → 业务地址 → 热池）任务
  3. 辅助运维做冷归集/热回笼的决策
- **架构前提**：每个商户拥有独立热池，不存在"全部商户"聚合视图
- **数据刷新**：页面加载时拉取一次，手动刷新浏览器更新（无自动轮询、无 WebSocket）
- **链/币种**：当前仅支持 TRON-USDT（多链场景后续迭代）

## 2. 三条核心资金流

### 2.1 常规归集（业务地址 → 热池）

| 属性 | 说明 |
|------|------|
| 方向 | 商户业务地址 → 热池 |
| 触发方式 | 自动（达到归集阈值） |
| 执行方式 | 系统自动执行 |
| Ledger 参与 | 否 |
| 本页面关注 | 体现在"入热池"数据线 |

### 2.2 冷归集（热池 → 冷钱包）

| 属性 | 说明 |
|------|------|
| 方向 | 热池 → 冷钱包地址 |
| 触发方式 | 线上自动触发（余额 > HOT_MAX）；测试环境手动创建 |
| 执行方式 | 系统全自动执行（无需运维介入、无需 Ledger 签核） |
| Ledger 参与 | **否** |
| 拼单 | 系统自动将多个热池地址的归集合并为一笔任务 |
| 目标地址选择 | 从商户绑定的冷钱包地址池中选余额最少的地址 |
| 本页面关注 | 调度任务列表（冷归集类型） |

### 2.3 热回笼（冷钱包 → 业务地址 → 热池）

| 属性 | 说明 |
|------|------|
| 方向 | 冷钱包 → 商户业务地址 → 热池（间接路径，冷钱包不能直接到热池） |
| 触发方式 | 系统创建任务（热池余额 < HOT_MIN） |
| 执行方式 | 运维在 Smart App 确认后，用外部钱包手动从冷钱包转账到商户业务地址 |
| 到达热池 | 业务地址收到资金后，达到归集阈值自动归集到热池（Step 2 系统自动） |
| Ledger 参与 | **否**（系统外操作） |
| 系统监听 | 通过区块扫描检测到业务地址收款 |
| 本页面关注 | 调度任务列表（热回笼类型）+ KPI"余额不足"告警 |

**热回笼完整链路**：
```
热池余额 < HOT_MIN → 系统创建热回笼任务（含目标业务地址 + 建议金额）
→ Smart App 推送给运维 → 运维在 Smart App 确认知悉
→ 运维在 VOSS 后台用商户能量池委托 Energy 给来源冷钱包地址
→ 运维用外部钱包从冷钱包手动转账到目标业务地址
→ 系统扫描区块，检测到业务地址收到入款 → 热回笼任务标记已完成
→ 业务地址达到归集阈值，自动归集到热池（属常规归集，不再归属此任务）
```

**完成判定**：扫块检测到目标业务地址收到入款即标记"已完成"，后续资金从业务地址归集到热池属于常规归集流程。

## 5. 水位线模块（实时数据）

### 5.1 配置参数（每个商户独立配置，商户编辑页设置）

| 参数 | 含义 | 本页面是否可改 |
|------|------|--------------|
| HOT_MIN | 最低线，低于此值触发热回笼 | 否（只读展示） |
| HOT_TARGET | 目标线 = (HOT_MIN + HOT_MAX) / 2，冷归集和热回笼的双锚点 | 否（自动计算） |
| HOT_MAX | 最高线，超过此值触发冷归集 | 否（只读展示） |

#### 水位线推荐值

本页面提供 HOT_MIN 和 HOT_MAX 的推荐值，辅助运维配置：

**HOT_MIN 推荐公式**：
```
HOT_MIN = PeakOut_per_hour × H_min × Safety
```
- `PeakOut_per_hour`：过去 14~30 天的每小时出款峰值（取 P95）
- `H_min`：最小扛住时长，即即使完全无入款也能扛住的小时数（常用 6h）
- `Safety`：保险系数（常用 1.1 ~ 1.3）
- 直觉：能扛住 H_min 小时出款尖峰的钱

**HOT_MAX 推荐公式**：
```
HOT_MAX = PeakOut_per_hour × H_max × Safety
```
- `PeakOut_per_hour`：同最低线，过去 14~30 天的每小时出款峰值（取 P95）
- `H_max`：最大扛住时长（常用 24h），超过此时长的资金不应留在热池
- `Safety`：保险系数（同最低线）
- 直觉：最低线和最高线本质逻辑一致（峰值 × 时长 × 系数），区别仅在于"扛多久"——最低线保最小安全，最高线封最大上限

**HOT_TARGET**：自动取 (HOT_MIN + HOT_MAX) / 2，同时作为冷归集和热回笼的双锚点：
- 冷归集时，归集金额 = 余额 − HOT_TARGET
- 热回笼时，建议回笼金额 = HOT_TARGET − 余额

**页面展示方式**：
- 水位条上用虚线标记推荐位置
- 水位条下方折叠推荐面板（可展开查看公式和参数）
- 展开详情包含公式、计算过程、与当前配置的对比偏差
- 底部数据窗口显示 P95 出款峰值、时长参数、保险系数

### 5.2 水位条（可视化）

| 区间 | 范围 | 颜色 | 含义 |
|------|------|------|------|
| 危险区 | 0 ~ HOT_MIN | 红色斜线 | 余额不足，触发热回笼 |
| 安全区 | HOT_MIN ~ HOT_MAX | 绿色 | 水位健康，无需操作 |
| 超限区 | HOT_MAX ~ ∞ | 红色斜线 | 超上限，触发冷归集 |

- 三条标记线：最低线 / 目标线 / 最高线（各带数值标注）
- 当前余额圆点（颜色随所在区间变化，脉冲动效）
- 鼠标悬停余额圆点显示精确金额 tooltip

### 5.3 智能状态条

描述当前热池水位的调度状态，只有 3 种互斥状态。

#### 状态机（3 种状态）

| 状态 | 触发条件 | 样式 | 主数字 | hint |
|------|---------|------|--------|------|
| 余额不足 · 需热回笼 | HOT_BALANCE < HOT_MIN | 红色 | 低于最低线 {差额} USDT | 热池余额跌破最低线，系统已触发热回笼 · 回笼 {金额} USDT 至目标线 |
| 水位正常 | HOT_MIN ≤ HOT_BALANCE ≤ HOT_MAX | 绿色 | （不显示） | 热池水位在安全区间，无需操作 |
| 触发冷归集 | HOT_BALANCE > HOT_MAX | 橙色 | 超最高线 {超出额} USDT | 热池余额突破最高线，系统已自动触发冷归集 · 归集 {金额} USDT 至目标线 |

#### 触发逻辑

```
余额 > HOT_MAX → 触发冷归集，归集金额 = 余额 − HOT_TARGET
余额 < HOT_MIN → 触发热回笼，回笼金额 = HOT_TARGET − 余额
HOT_MIN ≤ 余额 ≤ HOT_MAX → 不触发任何调度
```

- HOT_TARGET 纯粹作为"回归锚点"，不参与触发判断
- 无需 BATCH_MIN（最低冷归集金额）和能量条件，触发逻辑极简

#### 已有任务提示

当水位正常但存在活跃任务时，hint 末尾追加 `（已有 X 条任务不受影响）`，避免运维误解为已有任务被取消。

#### 6.3 任务状态体系（7 种，与资金调度完全一致）

```
待处理 → 处理中 → 已发起 → 已确认 → 已完成
                                    ↘ 失败
（任何阶段均可 → 已取消）
```

| 状态 | tag 颜色 | 冷归集含义 | 热回笼含义 |
|------|---------|-----------|-----------|
| 待处理 | 灰色 | 任务已创建 | 任务已创建 |
| 处理中 | 蓝色 | 系统准备中 | 系统准备中 |
| 已发起 | 黄色 | 链上交易已广播 | 任务已派发到 Smart App |
| 已确认 | 绿色 | 链上确认（上链） | 运维确认知悉，去外部钱包转账 |
| 已完成 | 灰色 | 资金已到冷钱包 | 扫块检测到业务地址收款（后续自动归集到热池） |
| 失败 | 红色 | 链上执行失败 | 超时未到账/异常 |
| 已取消 | 灰色 | 任务取消 | 任务取消 |

#### 6.3a 任务防重复

- 同一商户、同一方向（冷归集或热回笼），**只允许 1 条活跃任务**（状态为待处理/处理中/已发起/已确认）
- 已有活跃任务时，即使余额仍超出阈值，系统不再创建新任务

#### 6.3b 任务不可逆

- 任务一旦创建就**执行到底**，不因后续水位变化自动取消
- 如需取消，运维手动操作（状态改为"已取消"）

#### 6.4 失败处理

- **不自动重试**
- 热回笼任务有**系统超时机制**（超时时长由后端配置），超时后自动标记失败
- 失败后 xbb 聊天软件通知运维
- 运维排查具体失败原因，协调技术团队处理
- 失败原因在任务详情 Drawer 的"失败原因"字段和操作日志中展示

#### 6.5 任务详情 Drawer

点击调度任务列表的"详情"链接，打开右侧 Drawer，标题为"资金调度 #N"。

**排版**：左右两列 key-value 列表（label 在左灰色背景，value 在右），与后台资金调度详情页一致。

#### 6.6 冷归集执行规则

| 规则 | 说明 |
|------|------|
| 执行方式 | 拼单冷归集（从多个归集地址汇集到冷钱包） |
| 地址优先级 1 | 已轮换过的归集地址优先 |
| 地址优先级 2 | 钱包余额大额优先 |
| 冷归集目标 | 商户绑定冷钱包地址池中余额最少的地址 |
| 冷归集金额 | HOT_BALANCE − HOT_TARGET（以 HOT_TARGET 为锚点，精确金额） |
| 地址上限 | 后端配置决定 |
| 防重复 | 同商户已有活跃冷归集任务时不再创建新任务（见 6.3a） |

#### 6.7 热回笼执行规则

| 规则 | 说明 |
|------|------|
| 建议回笼金额 | HOT_TARGET − HOT_BALANCE（以 HOT_TARGET 为锚点） |
| 来源冷钱包 | **系统不指定**，运维自行决定从哪个冷钱包转出（因转账在外部钱包手动操作） |
| 能量委托 | 运维在 VOSS 后台用商户能量池将 Energy 委托给来源冷钱包地址（每次转账前按需操作） |
| 目标地址 | 商户现有的业务地址 |
| 执行方式 | 运维用外部钱包手动从冷钱包转账到业务地址 |
| 到达热池延迟 | 需经过业务地址归集流程，非即时到达 |
| 防重复 | 同商户已有活跃热回笼任务时不再创建新任务（见 6.3a） |

## 9. 告警体系

### 9.1 页面内告警

| 触发条件 | 展示位置 | 展示方式 |
|---------|---------|---------|
| HOT_BALANCE < HOT_MIN | KPI"热池余额" + "距目标线" + "调度状态" | 红色数字 + "需热回笼" |
| HOT_BALANCE < HOT_MIN | 水位线状态条 | 红色"余额不足 · 需热回笼"状态 |
| HOT_BALANCE > HOT_MAX | KPI"距目标线" + "调度状态" | 橙色"冷归集" + 归集金额 |
| HOT_BALANCE > HOT_MAX | 水位线状态条 | 橙色"触发冷归集"状态 |
| 有失败任务 | 调度任务列表 | 红色 tag |

### 9.2 xbb 聊天软件告警

| 场景 | 触发条件 | 级别 | 消息内容模板 |
|------|---------|------|-------------|
| 余额不足 | HOT_BALANCE < HOT_MIN | 紧急 | ⚠️ 【商户名】热池余额不足！当前 {余额} USDT，低于最低线 {HOT_MIN}，差额 {差额} USDT，建议创建热回笼任务 |
| 超出上限 | HOT_BALANCE > HOT_MAX | 警告 | ⚠️ 【商户名】热池超出上限！当前 {余额} USDT，超过最高线 {HOT_MAX}，超出 {超出额} USDT，系统已自动触发冷归集 |
| 调度任务失败 | 冷归集/热回笼任务状态变为"失败" | 紧急 | ❌ 【商户名】{任务类型}任务失败！任务ID: {ID}，金额: {金额} USDT，失败原因: {原因}，请运维排查处理 |
| 余额恢复 | HOT_BALANCE 从 < HOT_MIN 恢复到 ≥ HOT_MIN | 信息 | ✅ 【商户名】热池余额已恢复，当前 {余额} USDT，高于最低线 {HOT_MIN} |

**推送规则**：
- 每种场景首次触发时推送，不重复推送同一状态
- 余额恢复通知仅在之前发过"余额不足"告警后才推送
- 冷归集成功不发通知（超出上限告警仅在首次超限时发送）

## 10. 地址池模型

### 10.1 地址池结构

| 属性 | 说明 |
|------|------|
| 池的本质 | 逻辑分组（类似目录），管理冷钱包地址 |
| 池与商户关系 | 一个池可以绑定多个商户 |
| 命名 | 自定义名称（如"华东入池"、"中部交叉"） |
| 整批替换 | 想换一批地址时，可直接换池 |

### 10.2 地址生命周期

```
Smart App 生成地址 → 导入系统(未验证)
→ 创建设备验证任务 → Ledger签核通过
→ 入池 → 同步余额
→ 用商户激活池发小额TRX激活 → 地址可用
```

### 10.3 签核器

- 系统中显示为"内网签核器"（如 com-p-01）
- 对应物理设备为 Ledger 硬件签核器
- 通过 Smart App 连接

## 12. 已移除/简化项（相比 v1.0）

| 项目 | 原状态 | 变更 | 原因 |
|------|--------|------|------|
| 页面名称 | 资金监控中心 | 改为"调度中心" | 更简洁，突出调度职能 |
| 原4种任务状态 | 等待能量/回储中/已完成/失败 | 改为7种（与资金调度一致） | 取代资金调度页面，状态需统一 |
| 回储任务列表 | 独立面板 | 合并为 Tab 区域 | 和周期明细共享 Tab 区域 |
| Drawer 任务列表 | Drawer 包含任务 | 移除 | 任务已在调度任务 Tab 常驻展示 |
| 重试按钮 | 失败任务点击重试 | 移除 | 失败需运维排查原因，不能简单重试 |
| "全部商户" | 曾有下拉选项 | 删除 | 每个商户独立热池，聚合无意义 |
| 强制/普通类型 | 两种类型 | 改为冷归集/热回笼 | 按实际业务区分 |
| 手动触发扫描 | 按钮在任务列表标题栏 | 移除 | 线上为自动触发 |
| "运营"角色 | 目标用户 | 改为"运维" | 实际使用者为运维团队 |
| Telegram 通知 | 告警渠道 | 改为 xbb 聊天软件 | 实际使用的通讯工具 |
| Tab 区域位置 | 底部（趋势图下方） | 上移到水位线下方 | 运维高频操作，减少滚动 |
| BATCH_MIN 参数 | 最低冷归集金额门槛 | 移除 | 改用 HOT_MAX 直接触发，无需中间门槛 |
| 能量参数 | ENERGY_THRESHOLD / PER_TX / SAFETY | 移除 | 不再作为触发条件 |
| 条件 Pills | 金额 + 能量两个条件标签 | 移除 | 触发逻辑简化为 MIN/MAX 判断，无需条件展示 |
| 6 种水位状态 | 余额不足/无需冷归集/未触发/待执行/已触发/超上限 | 简化为 3 种 | MIN/MAX 触发模型下只需 3 个互斥状态 |
| "下次冷归集" KPI | 展示 4 种冷归集条件状态 | 改为"调度状态"（3 种） | 统一展示冷归集和热回笼的触发状态 |
| 水位条预警区 | HOT_TARGET ~ HOT_MAX 橙色区间 | 合并为安全区 | MIN~MAX 全部为安全区间 |
| 冷归集 Ledger 签核 | 冷归集需 Ledger 签核确认 | 移除，改为全自动 | 冷归集是热池向冷钱包转账，无需 Ledger |
| 超限恢复 xbb 通知 | 冷归集完成后发"超限恢复"通知 | 移除 | 冷归集成功不需要发通知 |
