# 信用系统深度讨论 — 问题 / 建议 / 旧做法 汇总

## 话题 1：数值平衡

### 1.1 准时放币奖励过高


|         | 内容                                                                                     |
| ------- | -------------------------------------------------------------------------------------- |
| **问题**  | +500/笔 × 20 笔/天 = 10,000/天，单项就能覆盖 1 次判定发币处罚（-10,000），准时放币变成刷分通道                        |
| **旧做法** | 2024 版无此项；2025 v1.0 设为 ≤1 分钟 +1,000、1-5 分钟 +500、5-15 分钟 +200                           |
| **建议**  | 整体减半：≤1 分钟 +500、1-5 分钟 +200、5-15 分钟 +100。基本义务不应成为经验值主要来源。调整后日均正向经验从 ~16,100 降到 ~10,100 |


### 1.2 工单处罚固定值，大小额一视同仁


|         | 内容                                                                                                                                        |
| ------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 判定发币 -10k / 判定退币 -30k 是固定值，活跃卖家 1-2 天就回血。大额不发货和小额不发货同等处罚不合理                                                                               |
| **旧做法** | 2024 版统一 -3,000；2025 v1.0 改为 -10,000 / -30,000 固定值                                                                                        |
| **建议**  | 固定底线 + 金额挂钩：判定发币 = -10,000 - 金额×50、判定退币 = -30,000 - 金额×100。设上限 = 当前经验值×0.8，留申诉空间。示例：5,000 元交易判定退币 → -100,000；100,000 元交易判定退币 → -1,430,000 |


### 1.3 时间衰减三段太粗，90 天以上打 2 折太激进


|         | 内容                                                                                |
| ------- | --------------------------------------------------------------------------------- |
| **问题**  | 三段衰减（1.0/0.6/0.2），90 天以上直接 0.2 权重。活跃用户短期休假（如出差 1 个月）可能断崖降级                        |
| **旧做法** | 2024 版无衰减；2025 v1.0 三段（30天 1.0、90天 0.6、90天+ 0.2）                                  |
| **建议**  | 改四段：30天 1.0、90天 0.7、180天 0.4、180天+ 0.2。加入等级保底机制：衰减后经验值 ≥ 当前等级下限×0.7，最多降 1 级而非自由落体 |


### 1.4 正负经验同等衰减，处罚会"自然洗白"


|         | 内容                                                                                                                  |
| ------- | ------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 如果负向经验也按同样速度衰减，3 个月前坑过人的卖家自动"洗白"，惩罚失去意义                                                                             |
| **旧做法** | 2024 版无衰减；2025 v1.0 未区分正负衰减速度                                                                                       |
| **建议**  | 非对称衰减。正向（好事忘得快）：30天 1.0/90天 0.7/180天 0.4/180天+ 0.2。负向（坏事记得久）：90天内 1.0/90-365天 0.8/365天+ 0.5。类似央行征信逾期记录保留 5 年但不永久的逻辑 |

## 话题 2：工单处罚细节

### 2.1 "判定发币"计入 3 次永久停权，卖家会觉得不公平


|         | 内容                                                                                                                   |
| ------- | -------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 判定发币 = 卖家最终发了币（虽然被强制），交易是成功的。卖家可能觉得"我只是慢了/忘了，最后还是给了，凭什么算我一次？"但不计入的话，卖家会故意拖延试探买家是否放弃                                  |
| **旧做法** | 2024 版工单失败统一 -3,000，统一计入 3 次红线；2025 v1.0 区分了扣分（-10k vs -30k）但仍然统一计入                                                  |
| **建议**  | 改为"计分制"：判定退币 = 1.0 次，判定发币 = 0.5 次，累计 ≥ 3.0 次触发永久停权。效果：纯退币 3 次停权（不变），纯发币 6 次才停权（给粗心但非恶意的卖家更多容错），混合 2 退币+2 发币 = 3.0 停权 |


### 2.2 工单协商取消 = 0 分，会被利用来逃避处罚


|         | 内容                                                                                                                                              |
| ------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 卖家收到钱后 → 私下联系买家"你取消我退钱" → 双方协商取消 → 卖家 0 扣分。更恶劣：卖家威胁买家"你不取消我就不发"                                                                                 |
| **旧做法** | 2024 版无明确区分；2025 v1.0 设为协商取消 = 0 分                                                                                                              |
| **建议**  | 按"谁发起"和"是否已付款"区分。买家未付款时取消 = 双方 0 分。买家已付款 + 买家主动取消 = 卖家 0 分。**买家已付款 + 卖家发起取消 = 卖家 -5,000**。频率异常（30 天内 ≥ 3 次协商取消）= 卖家 -10,000/次。防止"协商取消"成为逃避处罚的通道 |


### 2.3 工单处罚缺少买家侧规则


|         | 内容                                                                                                                                     |
| ------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 当前方案主要聚焦卖家处罚，但买家也可能恶意：假装付款骗放币、恶意开工单骚扰卖家、假截图。买家"零成本开工单"会消耗大量客服资源                                                                        |
| **旧做法** | 2024 版仅有买家取消阶梯扣分（-7,000/-15,000/-30,000...），无工单侧处罚；2025 v1.0 同样未涉及买家工单处罚                                                               |
| **建议**  | 新增买家工单后果：判定发币（卖家被强制发）= 买家 +0（正常结果）。判定退币且查实买家未付款/假付款 = 买家 -30,000。判定退币且发现假截图 = 买家 -80,000 + 标记欺诈。买家发起工单被驳回（无效工单）= 买家 -5,000。给工单增加"发起成本" |

## 话题 3：反 Sybil 落地难度

### 3.1 设备指纹在不同签名方式下的采集能力


|         | 内容                                                                                                                                                                                                      |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | iOS 上 TestFlight 和企业签/个人签的设备指纹采集能力差异大。IDFA 被限制后（iOS 14.5+），设备指纹的唯一性和稳定性下降。Android 相对好但也有刷机/改机工具                                                                                                         |
| **旧做法** | 2024 版无设备指纹体系                                                                                                                                                                                           |
| **建议**  | 不依赖单一标识符，采用**复合指纹**：屏幕分辨率 + OS 版本 + 语言 + 时区 + 电池容量 + 存储空间特征 + 网络行为模式。复合指纹即使没有 IDFA 也能达到 ~85-90% 唯一性。剩余 10-15% 的模糊区间，配合行为信号（使用时段、交易模式、打字节奏）做二次判定。**关键：设备指纹定位是"弱信号"，单独不触发处罚，必须与其他维度（IP/银行卡/交易图谱）交叉命中才标记** |


### 3.2 同 IP 段阈值可能误杀正常用户


|         | 内容                                                                                                                                                                                                                        |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | "同 IP 段 ≥ 5 个账号活跃"的规则 —— 同一公司/学校/咖啡厅 WiFi 下可能有多个真实用户，直接标记为 Sybil 组会误杀。运营商 NAT IP 更极端，几百万用户共用同一个出口 IP                                                                                                                      |
| **旧做法** | 2024 版无 IP 维度检测                                                                                                                                                                                                           |
| **建议**  | IP 永远只做"弱信号"，**单独不触发任何处罚**。具体规则改为三层：(1) 已知运营商 NAT IP 段 → 完全忽略 IP 维度，只看设备+银行卡+行为；(2) 住宅/商业 IP 同 IP ≥ 5 账号 → 标记"待核实"但不扣分不降权，必须交叉命中设备或银行卡才升级为 Sybil 组；(3) 同 IP + 同设备指纹 + 互相交易 → 高置信度 Sybil，直接标记。总原则：**宁可漏过不可误杀，误杀比漏过的伤害大得多** |


### 3.3 银行卡 1:1 绑定在实际业务中是否可行


|         | 内容                                                                                                                                                                                                                                          |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | "同银行卡绑定 ≥ 2 个账号 → 标记"—— 家人共用银行卡、换手机号重新注册旧卡绑新号等场景会误杀。中国大陆一人多卡很常见，但一卡绑多平台账号确实可疑                                                                                                                                                               |
| **旧做法** | 2024 版无银行卡维度检测                                                                                                                                                                                                                              |
| **建议**  | 区分"绑定"和"收付款"两个维度。(1) 同银行卡绑定 ≥ 2 个活跃交易账号 → 标记为"待核实"+ 要求补充身份验证（人脸/视频），验证通过则解除标记；(2) 同银行卡在 ≥ 2 个账号上作为**收款账户**出现 → 直接标记 Sybil，因为收款账户实际上是在帮别的账号收钱，风险极高；(3) 换手机号重新注册的场景 → 支持"老账号迁移"流程，新号继承旧号的银行卡绑定，旧号自动注销。**银行卡是反 Sybil 最可靠的"强信号"，因为实名银行卡难以批量伪造** |

## 话题 4：新人沙箱与业务增长的平衡

### 4.1 新人 7 天内只能与 L3+ 交易，可能匹配不到卖家


|         | 内容                                                                                                                                                                                       |
| ------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 如果 L3+ 卖家只占全部卖家的 ~50%，新人能匹配到的候选池直接缩小一半。高峰期可能还好，凌晨/低谷时段候选池可能只剩几个人                                                                                                                         |
| **旧做法** | 2024 版无新人限制，所有用户同等对待                                                                                                                                                                     |
| **建议**  | 删掉"只能与 L3+ 交易"的硬限制。改用**金额上限**作为主要风控手段（已经有沙箱限额），对手方等级限制带来的匹配池缩小对新人体验伤害太大。替代方案：新人首 7 天的订单，系统自动**优先匹配** L3+ 卖家（软偏好），但如果 L3+ 无人在线则允许匹配 L2。L1 卖家仍然排除（L1 本身就不应该接一键买币单）。这样风控由"硬门槛"变"软偏好"，匹配池不缩小 |


### 4.2 沙箱对商户（B端）和个人用户（C端）是否需要区分


|         | 内容                                                                                                                                                                               |
| ------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 认证商家新入驻也要从 5,000 元/笔限额开始？认证商家已经通过了营业执照/保证金等企业级审核，5,000 元限额会直接劝退                                                                                                                  |
| **旧做法** | 2024 版无商户/个人区分                                                                                                                                                                   |
| **建议**  | 双轨沙箱。**个人用户沙箱**（不变）：≤7 天 5k/笔、7-30 天 2w/笔、30 天+L2 5w/笔。**认证商家快速通道**：通过商家认证后直接从 5w/笔起步（跳过 5k 和 2w 阶段），7 天无工单后解锁 20w/笔，30 天无工单后无限制。理由：认证商家已有企业实名 + 保证金，风险远低于个人新用户，限额过低 = 直接损失商户入驻率 |


### 4.3 沙箱限额对买家和卖家是否统一


|         | 内容                                                                                                                                                                                                |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 新买家风险 = 最多损失自己的钱（自损）。新卖家风险 = 可能骗别人的钱（损人）。两种风险本质不同，统一限额不合理                                                                                                                                         |
| **旧做法** | 2024 版无区分                                                                                                                                                                                         |
| **建议**  | 分角色设置沙箱。**新买家**：限额放宽到 2× 标准（≤7 天 1w/笔、7-30 天 5w/笔），因为买家的风险是"自己亏钱"，平台和其他用户的损失有限。**新卖家**：限额保持严格（≤7 天 5k/笔、7-30 天 2w/笔），因为卖家不发货 = 买家亏钱。同时新卖家首 10 笔交易强制快速放币（≤5 分钟），超时直接 -5,000 经验（比老卖家更严），用"试用期"筛选诚信度 |

## 话题 5：等级权益的实际效果

### 5.1 L1 一键买币流量权重 = 0，是否太极端


|         | 内容                                                                                                                                                                                                                                                   |
| ------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | L1 完全没有一键买币流量 = 软封禁。L1 包含两类人：(a) 信用真的差的老用户 (b) 刚注册还没升级的新人。对新人一刀切 = 0 流量会严重影响留存                                                                                                                                                                       |
| **旧做法** | 2024 版无流量权重区分                                                                                                                                                                                                                                        |
| **建议**  | L1 流量权重从 0 改为 **0.2×**。效果：L1 卖家不是完全没单，而是"明显比别人少"。同时区分"L1 新人"和"L1 降级户"：新注册 30 天内的 L1 用户享受新人扶持流量（保底 3 单/天），30 天以上仍在 L1 的才按 0.2× 正常执行。完整调整后的流量倍率表：L1=0.2×、L2=1.0×、L3=1.5×（原 2.0×，缩小 L2-L3 差距避免 L2 抱怨）、L4=3.0×（原 4.0×）、L5=6.0×（原 8.0×）。梯度更平滑，不会让某个等级"断流" |


### 5.2 "争议默认倾向"合规/法务能否接受


|         | 内容                                                                                                                                                               |
| ------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 等级权益表里明确写"L4 略倾向、L5 倾向" —— 这等于公开宣布"信用高的人在争议中有优势"。如果被截图/传播，可能引发公关危机。从法律角度，平台在争议调解中应遵循公平原则，显式偏向某方可能违规                                                              |
| **旧做法** | 2024 版无此设计                                                                                                                                                       |
| **建议**  | **从等级权益表中删除"争议默认倾向"这一行**。但在 AI 判定卡片的内部逻辑中保留"信用画像影响证据可信度权重"的设计（见 AI 判定卡片文档 4.2 节）。效果一样：高信用用户的证据在 AI 判定中获得更高可信度权重 → 实际上在同等证据下更有利。但这是后台算法逻辑，不是公开承诺的"权益"。**可以做不可以说** |


### 5.3 费率折扣的成本


|         | 内容                                                                                                                                                                                                                                      |
| ------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | L3 九折、L4 八折、L5 七折，折扣力度大。随着信用体系运行，L4/L5 占比会逐步上升（健康趋势），但这意味着平台手续费收入持续下降。假设 L4+L5 占比从 15% 涨到 30%，手续费收入可能下降 8-12%                                                                                                                           |
| **旧做法** | 2024 版无费率区分                                                                                                                                                                                                                             |
| **建议**  | 折扣力度减半，且设月度上限。调整后：L3=97 折（原 9 折）、L4=95 折（原 8 折）、L5=9 折（原 7 折）。每月折扣总额上限 = 500 USDT（L3）/ 2,000 USDT（L4）/ 5,000 USDT（L5），超出部分恢复标准费率。这样：(a) 折扣感知仍然存在（每月确实能省钱），(b) 平台收入影响可控（有上限），(c) 大户不会无限薅折扣。另一种思路：不用折扣，改用"返利积分"（可兑换平台内权益但不能提现），完全消除对营收的影响 |

## 话题 6：时间衰减的技术实现

### 6.1 每日全量重算的性能问题


|         | 内容                                                                                                                                                                                                                                                                                                              |
| ------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | "每天凌晨重新计算所有用户当前经验值" —— 如果用户量百万级，每个用户的所有历史事件都要遍历一遍，计算量巨大。假设 100 万用户 × 平均 500 条历史事件 = 5 亿次计算                                                                                                                                                                                                                      |
| **旧做法** | 2024 版无衰减，经验值只做累加（O(1) 增量更新）                                                                                                                                                                                                                                                                                    |
| **建议**  | 不遍历历史事件，改用**4 个桶（bucket）增量维护**。每个用户维护 4 个经验值桶：bucket_30d / bucket_90d / bucket_180d / bucket_old。每天凌晨 cron 只做"桶间迁移"：把超过 30 天的事件经验从 bucket_30d 挪到 bucket_90d，以此类推。加权总经验 = bucket_30d×1.0 + bucket_90d×0.7 + bucket_180d×0.4 + bucket_old×0.2。复杂度从 O(用户×事件数) 降到 O(用户)，百万用户几分钟跑完。每个桶只需存一个聚合数值（正向总和 + 负向总和），不需要保留明细 |


### 6.2 衰减后经验值与"等级公式"的关系


|         | 内容                                                                                                                                                                                                                                       |
| ------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 原公式 信用分=f(经验值) 是基于"经验值只增不减"设计的。引入衰减后，一个正常活跃用户可能看到"我没犯错但信用分下降了"，这违反用户对信用系统"做好事加分、做坏事扣分"的直觉预期                                                                                                                                              |
| **旧做法** | 2024 版经验值只升不降（除非处罚），用户预期"分数稳定向上"                                                                                                                                                                                                         |
| **建议**  | 采用**双轨制**。(1) 用户看到的"信用分" = max(衰减后计算值, 历史最高信用分 × 0.85)，保证用户看到的分数不会大幅波动；(2) 系统内部用的"实际信用分" = 严格按衰减计算，用于流量分配、撮合权重、费率等。用户看到的是"稍有滞后的平滑版本"，系统用的是"实时精确版本"。如果用户长期不活跃（>90 天无交易），用户看到的分也会缓慢下降（因为 0.85 保底会随时间逐步降低）。这样既保留了衰减的实际效果，又不会让用户产生"被系统坑了"的感觉 |


### 6.3 衰减导致等级频繁升降的用户体验


|         | 内容                                                                                                                                                                                                                                            |
| ------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **问题**  | 在等级边界的用户（如 L4 下限 200,000 附近），每天衰减后的微小波动可能导致"今天 L4 → 明天 L3 → 后天又 L4"的反复跳动。用户看到等级频繁变化会困惑甚至愤怒                                                                                                                                                    |
| **旧做法** | 2024 版等级只升不降                                                                                                                                                                                                                                  |
| **建议**  | **降级滞后机制（Hysteresis）**。升级 = 即时生效（经验值达到上一级阈值 → 立刻升级，用户高兴）。降级 = 需要连续 7 天经验值低于当前等级下限才执行。效果：在边界波动的用户，只要 7 天内有 1 天回到阈值以上就不会降级。真正持续不活跃/信用下降的用户，7 天后才降级，给了缓冲窗口。同时配合"等级保底"（话题 1.3），双重保护。另外在降级前 3 天，可以在"我的"页面静默展示提示："您的信用经验近期有所下降"（不说会降级，只暗示），给用户自救的机会 |
